# 文书写作 SOP 流程

> 统一法律文书写作与格式规范。所有规则优先级：内容红线 > 格式硬规则 > 风格偏好。

---

## 一、总则：写作原则

### 1.1 核心原则

- **不写空话**：不写正确的废话，不堆砌辞藻，强调信息密度
- **结论先行**：先结论 → 再分析 → 再依据 → 最后策略
- **实务取向**：像真实律师文书，兼顾法官阅读习惯和实务推进
- **准确精当优先**：准确、精当是目标，简短只是结果

### 1.2 语言风格

- 不要学生腔，要接近执业 10 年律师的真实文书表达
- 避免 AI 程式化措辞——过度对称、模板化、深度嵌套编号
- 写完标题自问：执业 10 年的律师会这样写吗？不会就重写
- 长文不碎片化，接近论文式连续表达，逻辑连续，减少机械标题感

### 1.3 标点符号（硬规则）

- 法律文书**必须全中文标点**：， 。 ： ； （） " " 、 ——
- **严禁英文标点混入**：`, . : ; () " " 、` 全部禁止
- 生成后必须检查确认无残留英文标点

## 二、案件分析框架（动笔前的固定动作）

所有案件分析优先采用以下框架：

### 2.1 必做五项

1. **案件事实**：核心事实梳理
2. **时间证据表**：时间线 + 对应证据节点
3. **争议焦点**：法律争议提炼
4. **救济路径**：可选救济方案
5. **最优救济策略**：推荐路径及理由

### 2.2 必要时增加

- 举证责任分配
- 程序风险
- 实体风险
- 对方抗辩预测
- 裁判倾向

### 2.3 动笔前双前置（硬规则）

> **norm check + case check 双前置。不是事后补救，是起手第一件事。**

**Norm check（规范核对）：**
- 用元典 `yuandian_law_vector_search` 检索相关法条
- 对劳动争议等新规密集领域，先查现行法律、司法解释和规范性文件，再用案例验证裁判趋势
- 涉及"是否还能主张XX"等规则问题，必须先做 norm check 再做 case check

**Case check（案例验证）：**
- 用元典 `yuandian_case_vector_search` 检索相关案例
- 至少覆盖：同案由/相近案由、核心争议焦点、关键事实模式、上级法院或近三年优先
- 检索结果区分：核心类案、重要参考案例、反向案例

**输出要求**：norm check + case check 结果输出沉淀笔记后，再动笔写文书。

## 三、法律文书 Markdown 结构模板

### 3.1 通用结构

```
frontmatter
  ↓
标题（文书类型）
  ↓
当事人信息（“原告”、“被告”等当事人地位加粗）
  ↓
案由（若有）
  ↓
诉讼请求 / 申请事项 【居中】
  ↓
事实与理由（中文数字章节 + 加粗子标题）【居中】
  ↓
结语
  ↓
此致 XX法院 / XX机关
  ↓
具状人 + 日期
  ↓
法律依据（实体法 / 程序法分列，法条用 blockquote）
  ↓
证据清单（法院式六列表格）
```


### 3.2 行政起诉状专用结构

> 详见 memory `reference_complaint_format.md`。以下为关键参数：

- **Markdown 结构**：同上通用结构
- **案例引用**：`> [!law]- 案号 — 总结` 折叠 callout
- **文件命名**：`{客户简称}诉{被告机关}-{文件类别}-v{N}.ext`

### 3.3 当事人信息格式（硬规则）

- **原告、被告、申请人、被申请人、第三人**等当事人标签、地位**必须加粗**
- 当事人信息是法官/审查人员第一眼锚定的核心要素，加粗赋予视觉优先级
- 格式示例：

> **原告**：**张三**，男，1990年1月1日出生，汉族，住北京市朝阳区XX路XX号，身份证号：110101199001010000。

> **被告**：**北京XX科技有限公司**，统一社会信用代码：91110108XXXXXXXXXX，住所地：北京市海淀区XX路XX号。

- 当事人信息优先走 OCR 提取（身份证 → 百度 OCR → 文书字段），不肉眼读图猜字
- 材料包中可 OCR / evidence extraction 提取的金额、日期、债权凭证、付款方式等必须填入文书；仅缺失的当事人身份字段或无法核验字段留空

### 3.4 事实与理由标题风格

- 使用中文数字章节（一、二、三……）
- 子标题加粗，避免 AI 程式化措辞——不要过度对称、模板化
- 写完标题自检：执业律师会这样写吗？不会就重写

### 3.5 结语与落款

- 结语简洁有力，不啰嗦
- 此致：左对齐，写明法院/机关全称
- 具状人：右对齐，署名 + 日期（署名空着，由当事人自己写）

## 四、法律依据写作规范

### 4.1 法条引用格式

- 法条用 blockquote `>` 缩进引用
- 短条文（1-2 行）：写完整条文原文
- 长条文（3 行以上）：写关键节选，不能完全不写条文内容
- 笔记中长法条用 `> [!law]- 法规名称 + 条文号` 折叠 callout

### 4.2 法条架构排查（硬规则）

写起诉状时必须按以下五段式排查法条架构：

1. **基础义务及请求权基础**：被告应履行什么义务（如承揽人交付义务 §780），请求对方履行的请求权基础是什么
2. **违约责任**：违约的 base 条款（如 §577）
3. **解除权**：解除合同的依据
4. **解除效果**：解除后的法律效果（如恢复原状 §566）
5. **损失计算**：损失计算或赔偿依据

每一条法条必须逐条用元典 `yuandian_rh_ft_detail` 核对 wording，验证 trigger 是否对得上案件 fact pattern。

### 4.3 法律依据元典核对（硬规则）

> **任何法律依据不得默认真实。当事人或他人提供的引用更需逐条核对。**

**核对四要素**：
1. 法规名称是否正确
2. 条文号是否存在
3. 条文内容是否与原文一致
4. 法规是否现行有效

**重点怀疑对象**：部门规章、规范性文件（当事人最常编造这一层），其次是司法解释。基础法律（行诉法、民法典、刑法等）相对可靠，但条文号、款项编号仍可能写错。

**核对结果处理**：
- 虚假引用标记 ❌ 并给出正确的替代条文
- 此规则优先级高于"默认采信用户提供材料"

**禁止事项**：
- ✗ 不得在提交给法院/当事人的正式文书中写"以下法律依据均已通过元典核对"——核对是 AI 内部保障流程，不应出现在对外文书中

## 五、证据清单规范

### 5.1 表格格式（硬规则）

证据清单**默认采用法院式六列表格**：

| 编号 | 证据名称 | 份数 | 页数 | 原件或复印件 | 证明对象 |

- 表格标题为"证据清单"，居中加粗
- 表头完整，不再使用简化三列表格
- 仅有复印件的在"原件或复印件"列写"复"

### 5.2 证明对象写作要求（硬规则）

- **必须写成完整事实句**，格式为"证明……"
- 不要只写抽象词（如 ✗"证明合同关系" ✗"证明付款事实"）
- 应尽量包含：关键时间 + 主体 + 金额 + 行为 + 法律意义
- 正确示例：✓"证明2022年12月22日，原告按被告要求向被告开具四张发票290669.75元（税率13%）"

### 5.3 证据清单内容红线

- 只列实际存在的证据材料
- 不得编造"计划取得"的材料
- 无法确认的证据标注"待补"或直接不列

---

## 六、附件结构规范（硬规则）

### 6.1 附件顺序

统一为：

> **附件一：证据材料**
> **附件二：法律依据**

- 证据优先，法律依据在后
- 不得再写成附件一法律依据、附件二证据材料

### 6.2 附件内部编号

附件标题必须写全称（"附件一：证据材料"、"附件二：法律依据"），附件内部每项仍需编号：

```
附件一：证据材料

1．申请人身份证复印件。
2．借款合同原件。
3．银行转账凭证复印件。
```

- 避免的是机械堆砌"复印件一份"式空话，不是取消附件内部编号
- 编号方便接收机关核对和装订

### 6.3 法律依据附件要求

- 不能只列法规名称和条号
- 短条文（1-2行）：必须写完整条文原文
- 长条文（3行以上）：可以不写全文，但必须写关键节选
- 不能完全不写条文内容

## 七、不同文书类型特殊规则

### 7.1 监督求助文书（纪委监委 / 检察院）

> 向纪委监委、检察院等监督部门提交的求助书/举报书，本质是**提供证据线索和事实线索，而非作出法律裁决**。

**"涉嫌"规则（硬规则）**：
- 所有对他人行为违法性的确定性判断，一律改为"涉嫌"
- ✗ "该行为违反…" → ✓ "该行为涉嫌违反…"
- ✗ "构成程序违法" → ✓ "涉嫌构成程序违法"
- ✗ "故意伤害"（作为结论）→ ✓ "涉嫌故意伤害"

**违禁词黑名单**：
- 保护伞 → 删除或用描述性语言替代
- 滥用职权 → 删除或用描述性语言替代
- 滥权行为 → 删除或用描述性语言替代
- 系统性违法 → 删除或用描述性语言替代
- 视而不见 → 删除或用描述性语言替代

**核心姿态**："请贵机关查一查这些事是否存在问题"，而非"这些事就是违法的"。

### 7.2 起诉状

- 必须完成 norm check + case check 双前置（见 §2.3）
- 法条五段式排查（见 §4.2）
- 诉讼请求逐项编号，每项独立、明确、可执行
- 事实与理由按时间线展开，每个关键事实对应证据清单中的证据

### 7.3 答辩状

- 逐项回应原告诉讼请求和事实主张
- 对原告每项事实主张明确：承认 / 否认 / 不清楚
- 积极抗辩 + 消极抗辩分列

### 7.4 代理词 / 法律意见书

- 案件事实陈述后直接进入法律分析
- 法条引用后紧跟类案裁判观点
- 预测法院可能的裁判倾向及风险

### 7.5 保全申请书

- 明确保全标的（账号、房产、股权等）
- 账号等不确定信息写"以XX查询为准"，不得自行补全
- 提供担保方式及金额

### 7.6 执行异议申请书

- 明确执行依据（判决书/调解书/公证债权文书）
- 异议事由逐项列明，附对应证据

## 八、格式排版规范

### 8.1 字体字号（通用法律文书）

| 元素    | 字体  | 字号       | 样式           |
| ----- | --- | -------- | ------------ |
| 大标题   | 黑体  | 16pt（三号） | 居中           |
| 章节标题  | 黑体  | 14pt（四号） | 加粗           |
| 子标题   | 黑体  | 12pt（小四） | 加粗           |
| 正文    | 仿宋  | 12pt（小四） | 首行缩进 2 字符    |
| 法条引用  | 仿宋  | 11pt     | 左缩进 1cm      |
| 当事人信息 | 仿宋  | 12pt     | 标签加粗（原告/被告等） |

### 8.2 裁判文书特殊规范

| 项目 | 规范 |
|------|------|
| 标题 | 方正简体 / 小标宋体 |
| 一级标题 | 黑体三号 |
| 正文 | 仿宋三号 |
| 上边距 | 3.7cm |
| 下边距 | 3.5cm |
| 左边距 | 2.8cm |
| 右边距 | 2.6cm |
| 行距 | 1.25 倍 |

### 8.3 页边距与行距（通用）

| 参数 | 默认值 | 可调整范围 |
|------|--------|-----------|
| 上边距 | 2.54cm | 通常不动 |
| 下边距 | 2.54cm | 通常不动 |
| 左边距 | 3.18cm | 可调至 2.8cm |
| 右边距 | 3.18cm | 可调至 2.4cm |
| 行距 | 1.25 倍 | 不超过 1.5 倍 |

### 8.4 数字金额格式（硬规则）

- **不加千位分隔符（逗号）**：写"68000元"而非"68,000元"
- 小数点后保留两位不受影响（如 20800.00）
- 罚款金额在笔记/表格中可用橙棕加粗标记

### 8.5 排版技巧

- 善用分层标题
- 善用加粗、下划线
- 善用表格进行内容归纳整理（表格可使文档更加规整）

## 九、Word 生成技术规范（硬规则）

### 9.1 样式清理（强制执行）

生成 docx 时，**必须显式清空文档模板中全部以下样式**，放在任何内容写入之前：

- Heading 1–9 样式
- Heading 1–9 Char 样式
- TOC Heading 样式

**清理操作**：将以上样式的 `theme_color` 设为 None，`font.color.rgb` 设为 `RGBColor(0,0,0)`。

不依赖 Word 默认样式不染色的假设——这是 python-docx 生成脚本中的固定模板清理步骤。

### 9.2 禁止的 Word 样式

- **严禁使用 Word 内置蓝色"标题1/Heading 1"等一眼 AI 味的标题样式**
- 法律文书标题必须使用朴素黑体/仿宋格式
- 黑色字体，不带蓝色、不带下划线、不带主题色
- 章节标题可用黑体加粗，但必须像真实律师文书，不得出现 Word 默认样式痕迹

### 9.3 输出默认

- 法律文书默认直接生成 .docx 文件，不再单独说明
- 生成脚本保留，以便后续更新重新生成

## 十、内容红线（优先级最高）

### 10.1 不编造证据材料 — 任何数字和事实必须可追溯

> 源起：关怀民案保全申请书 v2 将"账号以银行查询为准"擅自替换为具体账号 6222 6204 1000 4235 206，该账号在任何原始材料中均不存在。

**根因**：AI 的"习惯性补全"倾向——遇到"以XX查询为准""待补"等不确定表述时，默认把它们变成"确定的具体内容"。在法律场景中这不是 helpful，是 dangerous。

**铁律四条**：

1. **全面追溯原则**：凡是写入文书的具体信息——账号、金额、日期、身份证号、统一社会信用代码、**以及任何事实性陈述**（某人在某时某地做了某事、某笔款项的用途、某次沟通的内容、某事件的起因经过等）——写入前必须能回答"这个信息来自哪个原始材料？"。不能回答 = 不能写

2. **不确定信息留空，不补全**："以银行查询为准"就是"以银行查询为准"，绝不能自动升级成具体数字；事实叙述中的空白同样不得用合理推测填补。模糊是 feature，不是 bug

3. **区分原文与推理**：文书中的陈述必须能区分（a）直接来自原始材料的事实 和（b）基于事实的法律推理/论证。前者不得编造，后者必须有前者支撑

4. **自检**：每次生成文书后，逐项过一遍关键数字和事实性陈述，确认每个都有原始材料出处。无法确认的标注"待补"或直接不列

**"不编造证据材料"完整定义**：
- 法律文书中的所有具体信息（数字、事实性陈述、证据名称、当事人行为描述、时间线节点等），必须有原始材料支撑
- 不得从既有文书中照抄证据清单和数字，逐一核验原始材料
- 无法确认的证据/数字/事实，标注"待补"或直接不列
- "其他信息空着"一类指令应理解为**无法从材料中确认的信息留空**，不是把材料包里可 OCR 提取的金额、日期、付款方式、债权凭证等事实也留空；生成文书前必须先对材料包 OCR / evidence extraction，只空缺失的当事人身份字段或无法核验字段
- 此规则优先级高于"让文书看起来完整"——宁可留白，不可编造

### 10.2 当事人信息 OCR 优先

- 身份证图片不靠 AI 肉眼读——精度不够，生僻字和身份证号容易错
- 默认走百度 OCR 身份证结构化接口
- 流程：`~/Desktop/LIBRARY/ocr_idcard.py auto <图片路径>` → 提取 `parsed` 字段 → 填入文书当事人信息段落
- 批量场景：`~/Desktop/LIBRARY/ocr_idcard.py batch <文件夹> -o results.json` → 逐条灌入文书模板

### 10.3 法律依据和类案必须元典核对

> 双核对：法律依据 + 类案裁判规则，缺一不可。

- 收到任何含法律依据的材料，必须用元典逐条核对四要素（见 §4.3）
- 写任何法律文书时，除元典核对法律依据外，还必须用元典检索相关案例
- 类案核对四要素：案号/法院/裁判日期真实 + 裁判观点来自原文 + 与本案事实和争议焦点相似 + 是否存在反向或区别案例
- 文书正文未必都要引用案例，但内部起草、案件沉淀、策略判断必须完成 case check
- 如时间或材料不足未做，必须明示"类案检索未完成"
- 对称规则：自己不编证据材料，也不采信他人编的法律依据或案例规则

## 十一、文件命名与版本管理

> 详见 02-04 §8 文件命名规范。以下为摘要。

### 11.1 双层命名体系

| 层级 | 格式 | 用途 | 示例 |
|------|------|------|------|
| 项目级（文件夹名） | `{前缀}-{YYMMDD}-{序号}-{ShortName}` | 内部管理 | `S-260624-001-陈玉欣诉周万略健康权纠纷` |
| 文件级（交付文件） | `{案名简称}-{文书类型}-v{N}` | 客户可见 | `陈玉欣诉周万略-民事起诉状-v1.docx` |
| 内部工作文件 | `{编号}-{描述}-v{N}` | AI工作底稿 | `L-01-01-民事起诉状-v2.md` |

### 11.2 版本与语种

- 修订版本后缀：`v1`、`v2`……
- 英文文本：加 `（en）`
- 中英对照：加 `（cnen）`

### 11.3 三类交付版本

| 版本 | 英文 | 用途 | 特征 |
|------|------|------|------|
| 清洁版本 | Clean Version | 最终交付 | 所有修订痕迹已清除，整洁正式 |
| 修订版本 | Redline Version | 内部审阅 | 保留删除线、插入线等修改痕迹 |
| 编辑版本 | Edits Version | 变更说明 | 仅含修改内容列表 |

## 十二、文书生成完整标准流程

> 每一步必须按顺序执行，不可跳过或颠倒。

### Step 1：材料消化

- 读取用户提供的全部材料（文书、合同、证据图片、判决书等）
- 对图片/扫描件做 OCR 提取
- 对当事人身份证走百度 OCR 身份证接口
- 整理时间线 + 事实节点 + 证据对应关系

### Step 2：Norm Check（规范核对）

- 确定案件涉及的法律关系、请求权基础
- 用元典 `yuandian_law_vector_search` 检索相关法条
- 逐条核对四要素（名称、条号、内容、现行有效）
- 劳动争议等新规密集领域**先 norm check 再 case check**

### Step 3：Case Check（案例验证）

- 用元典 `yuandian_case_vector_search` 检索类案
- 提炼裁判观点、法院说理、主流规则和反向风险
- 区分核心类案 / 重要参考案例 / 反向案例
- 输出沉淀笔记

### Step 4：案件分析

- 按 §2 框架完成案件分析（事实、时间证据、争议焦点、救济路径、最优策略）
- 必要时增加举证责任、程序风险、对方抗辩预测

### Step 5：撰写 Markdown 文书

- 按 §3 结构模板逐段撰写
- 全中文标点
- 法条按 §4 五段式排查
- 数字/事实逐项确认可追溯（§10.1 铁律）

### Step 6：证据清单

- 按 §5 六列表格整理
- 证明对象写完整事实句
- 逐项核对证据来源，不编造

### Step 7：自检

- 标点符号全中文
- 关键数字和事实逐项确认出处
- 法条 wording 已元典核对
- 当事人信息已 OCR / 已标注待补
- 附件顺序正确（证据在前，法律依据在后）

### Step 8：生成 Word

- 按 §9 技术规范生成 .docx
- 先清理样式 → 再写入内容
- 标题黑色、无主题色、无蓝字

### Step 9：案件复利沉淀

- 将本案 norm check + case check + 裁判规则 + 实务启发整理进案件沉淀文档
- 见 [[案件复利沉淀]] 机制

## 附录 A：笔记格式化体系（法律实务笔记专用）

> 仅施用于法律实务类笔记（合规手册、案例整理、合同审查清单等）。原始协议全文、法条原文等引用材料不施加格式化。

### 配色语义表

| 元素 | 样式 | 含义 |
|------|------|------|
| 法条名称+条文号 | `<span style="color:#C41E3A; font-family: KaiTi, STKaiti, 楷体, serif;">` 红楷体 | 法律渊源锚点 |
| 段内子标题/关键概念 | `<span style="color:#1a6e9e; font-weight:bold;">` 深蓝加粗 | 实务操作条目、审查要点名 |
| 关键词高亮 | `<span style="background:#fff3cd; padding:1px 3px;">` 黄底 | 核心术语、系统名、触发条件 |
| 高危结论/风险提示 | `<span style="color:#B36B00; font-weight:bold;">` 橙棕加粗 | 风险结论、禁止性提示（红色不得用于非法条内容） |
| 合规/正向 | `<span style="color:#2b6e2b; font-weight:bold;">` 绿加粗 | 合规通过标记、最佳实践 |
| 罚款金额 | `<span style="color:#B36B00; font-weight:bold;">` 橙棕加粗 | 表格中的处罚金额数字 |

### 文字修饰语义表

| 修饰 | 用法 | 含义 |
|------|------|------|
| 加粗 | `**text**` | 强调关键结论、义务类型名 |
| 下划线 | `<u>text</u>` | **绝对红线**——不可做、不可信的硬性禁止事项 |
| 斜体 | `<em>text</em>` 或 `*text*` | 法律命题表述、区分性解释、场景化举例、背景铺垫 |
| 黄底荧光 | `<span style="background:#fff3cd;">` | 需要一眼锚定的核心术语、系统/平台名 |

### Callout 语义表

| Callout | 用途 |
|---------|------|
| `> [!important]` | 重点审核板块、核心结论 |
| `> [!warning]` | 风险提示、合规底线、高危风险提示 |
| `> [!tip]` | 实务总结、操作建议 |
| `> [!law]-` | 超过3行的法条全文，默认折叠 |

### 红色专用规则

**红色仅用于法律、法条、法源锚点**（法律名称、条文号、法规依据）。风险结论、禁止性提示、罚款金额等一律使用橙棕/蓝/黄/绿，不使用红色。

### 禁止使用的格式

- 不用 emoji 标题（除非法律案例标记 🏛️）
- 不用 `<kbd>` 或 `<code>` 标记法律术语
- 不用彩色背景大块段落（仅限 inline 关键词高亮）
- 不用多种颜色混在同一 span 中（一个 span 只做一件事）

### 默认执行规则

法律实务类笔记的"整理"默认包含两件事：
1. **structure cleanup**：标题层级、表格、callout、章节归位
2. **visual signals**：红楷体法条、蓝粗概念、黄底关键词、橙棕风险、绿粗合规路径、下划线红线、斜体区分/举例

不得只做 heading/table cleanup 后就结束。

## 附录 B：法院案例原文保留规则

> 来源：元文档 §4 案例笔记整理规则。此规则优先级高于"简洁"原则。

- 对"法院认为"、"裁判观点"、"律师点评"及法院、律师其他精彩说理段落，必须**原文保留**，不可删减概括
- 法院的论证逻辑、价值判断、裁判理由、政策考量等精华表述，是判例笔记最有价值的 material
- 整理判例时，优先提取并原样保留这些段落；宁可保留冗余，不可丢失精华
- 对法院特别精彩、有代表性的表述，应标注为 highlight，方便后续检索引用
- 精彩程度不亚于核心段落的法院附带论述（obiter dicta），同样保留

**处理方式**：
1. 识别判例中的核心段落
2. 原文完整提取，放显著位置
3. 必要时在原文前后加注解读/批注，原文本身不删不改
4. Obiter dicta 同样保留

## 附录 C：合同审查输出框架

合同审查不仅看条款合法性，还必须分析以下维度：

1. 合同真实目的
2. 项目背景
3. 商业结构
4. 权责是否匹配
5. 风险是否可控
6. 验收是否可执行
7. 付款条件是否锁定
8. 违约责任是否对等
9. 是否存在隐性义务
10. 是否存在监管风险

**输出方式**：条款级修改意见 + 风险级分析 + 商业逻辑分析 + 补充协议建议
