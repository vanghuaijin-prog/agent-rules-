# 桌面档案管理工作流 v2.0

> 覆盖桌面文件从接收到归档的全生命周期。v2.0核心变更：引入Library层、双时间点自动扫描、AI主动处理机制、Active超期自动归档。

## 1. 桌面四层结构

```
~/Desktop/
├── Inbox/          ← 缓冲区，接收所有新材料
├── Active/         ← 进行中的案件/项目
├── Archives/       ← 已结案/归档/结项（按时间归档）
└── LIBRARY/        ← 参考资料（科技、商学、人文、个人信息等）
```

**铁律：** 桌面根目录不得出现散落文件。所有新材料一律先进 Inbox，由AI在扫描时点统一编号并分流。

### 1.1 各层职能

| 层 | 职能 | 进入条件 | 离开条件 |
|----|------|----------|----------|
| Inbox | 缓冲区，待首次处理 | 用户扔入任何新材料 | AI扫描处理后移出 |
| Active | 进行中的案件/项目 | AI完成首次处理（赋ID+有MD指令） | 超4天无变动→自动归档 |
| Archives | 已结案/已结项 | Active超期归档 / 用户手动归档 | 一般不离开 |
| LIBRARY | 非案件类参考资料 | AI识别为非法律工作材料 | 一般不离开 |

### 1.2 Archives内部结构

按时间归档，不按案由：

```
Archives/
├── 2026/
│   ├── 06/
│   │   ├── S-260624-001-陈玉欣诉周万略健康权纠纷/
│   │   └── C-260624-001-XX合同/
│   └── 07/
└── 2027/
```

**理由：** 能进Archives的说明已完成使命，按时间归档零歧义。需要按案由找时，从Active近期项目或Obsidian索引回溯。

### 1.3 LIBRARY内部结构

按学科/领域分类：

```
LIBRARY/
├── AI学习/
├── 商业与科技/
├── 人文与发展/
├── 软件/
├── wallpaper/
└── （按需扩展）
```

**铁律：** 从 Inbox 移入 LIBRARY 时，必须根据文件内容自动归入对应子目录，严禁散落在 LIBRARY 根目录。

### 1.4 LIBRARY子目录分流规则

| 文件类型/内容 | 归入子目录 | 判断依据 |
|--------------|-----------|---------|
| 编程脚本/开发工具/技术文档 | 软件/ | .py/.js/.sh等代码文件，或内容涉及编程 |
| 思维方法/沟通技巧/人文社科 | 人文与发展/ | 心理学、哲学、沟通、个人成长类 |
| 商业资讯/科技动态/行业分析 | 商业与科技/ | 商业新闻、科技产品、行业报告 |
| AI相关学习资料 | AI学习/ | 大模型、机器学习、AI工具教程 |
| 图片/壁纸/视觉素材 | wallpaper/ | 图片文件、桌面背景类 |
| 无法判断类型 | 留在Inbox标记"待分类" | 等用户确认 |

## 2. 双层自动维护机制

桌面自动维护分为**轻量清理**和**深度处理**两层，前者零token消耗，后者真正执行Inbox中的任务指令。

### 2.1 轻量清理（本地脚本，每小时自动）

| 项目 | 说明 |
|------|------|
| 频率 | 每3小时一次（工作时间内） |
| 执行方式 | 本地Python脚本，无AI参与 |
| token消耗 | 零 |
| 动作 | (1) 桌面根目录散落文件 → 移入Inbox；(2) 已有ID的文件检查归位 |

**铁律：** 轻量清理只做文件搬运，不做任何内容处理。

### 2.2 深度处理（AI驱动，每日 11:00 + 22:30）

| 时间 | 说明 |
|------|------|
| 每日 11:00 | 处理上午及之前积累的材料 |
| 每日 22:30 | 处理全天材料，收尾当日工作 |

### 2.3 深度处理执行流程

```
深度处理触发（11:00 / 22:30）
│
├── Step 1: 轻量清理（桌面散落文件 → Inbox）
│
├── Step 2: 逐个处理Inbox
│   │
│   ├── 文件夹名以"T-"开头？
│   │   └── 是 → 模板，直接移入桌面模版库/，不赋ID
│   │
│   ├── 有MD指令 + "1"标记？
│   │   └── 是 → 读MD，按指令执行任务（生成文书/整理证据等）
│   │       └── 执行完成后 → 移入Active/
│   │
│   ├── 有MD指令但无"1"标记？
│   │   └── 赋ID + 标记"待处理"，说明理由
│   │
│   ├── 无MD？
│   │   ├── 含"1" → 赋ID，标记"待处理"等用户补MD
│   │   └── 不含"1" → 进入§4材料分流
│   │
│   └── 已处理过的（已有ID）→ 跳过
│
├── Step 3: 检查Active超期（详见§5）
│
└── Step 4: 追加行为日志到当周周报
```

**核心原则：** 深度处理 = 真正完成Inbox中的任务。有MD指令的必须执行，不是只标"待处理"。'扫描'必须等价于'处理完成'。

## 3. 案件/任务处理流程（文件夹名含"1"）

### 3.1 判断逻辑

```
文件夹名含"1" → 确认为案件/任务
│
├── 赋ID + 重命名（按§6规则）
│
├── 文件夹内有MD指令文件？
│   ├── 是 → 读取MD，按用户声明的流程和深度执行
│   │   └── 完成后 → 移入Active/
│   │
│   └── 否 → 标记"待处理"
│       └── 重命名：文件夹名最前加"待处理-"前缀
│       └── 留在Inbox，等下一扫描时间点再识别
```

**核心原则：** MD即指令。用户在MD中写明要做什么、走什么流程、走多深，AI照做即可，无需维护关键词映射表。

### 3.2 MD指令文件格式

MD文件放在案件文件夹内，格式与"随取所用"模板一致：

```markdown
## 做什么:
（一句话说清任务）

---
（粘贴素材、对话记录、需求原文等）
```

### 3.3 当次无法完成的案件

如果MD指令存在但任务当次无法完成（如需大量研究、等法院信息等）：
- 仍移入Active/
- 在MD中标注当前进度
- **Inbox只保留"还没开始第一次处理"的**

## 3.4 PDF排序规则

### 3.4.1 合并时排序

多个PDF合并时，按文件mtime（修改时间）升序排列，即按用户移入的先后顺序合并。

### 3.4.2 页面重排

处理案件时读取PDF内容，如发现页面乱序，在处理过程中按内容逻辑重新排列页码顺序。不单独开任务做页面排序。

## 4. 非案件材料分流（文件夹名不含"1"）

### 4.1 分流规则

```
扫描Inbox中的非"1"标记文件夹
│
├── 法律相关但非紧急案件材料
│   └── 赋ID + 重命名 → 留在Inbox等用户指令
│
├── 科技动态/经济学/商学/文学/个人信息等
│   └── 移入LIBRARY/对应子目录/
│
└── 无法判断类型的
    └── 赋ID + 标记"待分类" → 留在Inbox
```

## 5. Active超期自动归档

### 5.1 规则

Active内文件夹**超过4天无任何变动**（无文件修改、无新增），AI自动将其移入Archives。

### 5.2 检测方式

使用文件系统mtime，零token消耗：

```bash
find ~/Desktop/Active -mindepth 1 -maxdepth 1 -type d -mtime +4
```

### 5.3 归档执行

1. 将超期文件夹移入 `Archives/{YYYY}/{MM}/`
2. 记录到清理日志
3. ID不变，仅位置变更

### 5.4 豁免

用户可在文件夹内放置MD标注"长期活跃"，跳过自动归档检测。

## 6. 全局文件ID分配

### 6.1 ID格式

`{前缀}-{YYMMDD}-{序号}-{ShortName}`

### 6.2 前缀映射

| 类型 | 前缀 | 示例 |
|------|------|------|
| 诉讼案件 | S- | S-260624-002-李波诉迟成追偿纠纷 |
| 合同/法律意见/合规 | C- | C-260624-001-XX公司服务合同 |
| 研究 | R- | R-260624-003-证据整理SOP-v2 |
| 商业项目 | B- | B-260624-001-极致垂钓商业计划 |
| 个人信息 | P- | P-260624-001-个人证件材料 |
| 模板 | T- | T-文书模板（直接入模版库，不赋ID） |
| 其他 | X- | X-260624-001-会议记录 |

### 6.3 命名规则

- 年份必须准确对应实际年份（2026年写作'26'）
- 计数器严格按日期和类型双重维度独立归零
- ShortName必须具备自解释性：
  - 诉讼案件 = 原告 + 被告 + 案由
  - 非案件类 = 准确描述性名称

## 7. 案件文件夹结构

诉讼案件进入Active后，根目录结构遵循SOP-v2 §4.1：

```
S-{YYMMDD}-{序号}-{案名}/
├── P1-原告(姓名)/        ← 原告方材料
│   ├── 01-当事人身份材料/
│   ├── 02-XXX/            ← 按实际证据顺序编号
│   └── 03-XXX/
├── D1-被告(姓名)/        ← 被告方材料
├── T1-第三人(姓名)/      ← 第三人材料（如有）
├── L-律师工作成果/        ← AI生成的文书、分析报告
│   ├── 01-文书/
│   └── 02-分析/
├── M-证据映射表/          ← 映射表 + 证明组索引
└── Q-待补充队列/          ← 缺失材料占位
```

**大类编号规则（SOP-v2 §3.3 v2.1.0修订）：**
- 01固定 = 当事人身份材料
- 其他大类从02开始**按实际材料顺序编号**
- 参考表中的编号仅为示例，禁止跳号

## 8. 行为日志（周报制）

行为日志采用**每周合并**记录制，存放于 `行为日志/` 文件夹。

### 8.1 日志周期

- 每周一生成新周报文件，文件名格式：`行为周报-{YYYY}-{WW}.md`（WW为ISO周数）
- 每日扫描的操作记录追加到当周周报中
- 周报覆盖范围：周一至周日

### 8.2 日志格式

```markdown
# 行为周报 {YYYY} 第{WW}周 ({MM-DD} ~ {MM-DD})

## {YYYY-MM-DD} 扫描记录

### 桌面根目录清理
| 原文件名 | 操作 | 目标位置 |
|----------|------|----------|
| xxx.docx | 移入Inbox | Inbox/ |

### Inbox处理
| 文件夹 | 操作 | ID | 目标位置 | 备注 |
|--------|------|-----|----------|------|
| 1-张三诉李四 | 赋ID+处理 | S-260626-001 | Active/ | 有MD，起诉状 |
| 待处理-王五案 | 标记待处理 | S-260626-002 | Inbox/ | 无MD |
| 科技文章 | 移入Library | — | LIBRARY/商业与科技/ | 非案件材料 |

### Active超期归档
| 文件夹 | 最后修改 | 目标位置 |
|--------|----------|----------|
| S-260620-001-... | 2026-06-20 | Archives/2026/06/ |

---

## {YYYY-MM-DD} 扫描记录
（次日记录追加在此）
```

## 9. 检查清单

- [ ] 含"1"的文件夹已赋ID并检查MD
- [ ] 有MD的案件已按指令处理并移入Active
- [ ] 无MD的案件已标记"待处理"
- [ ] 非案件材料已移入LIBRARY
- [ ] Active超期文件夹已归档

### 案件启动检查清单

- [ ] 已读取元文档和相关SOP

- [ ] P1/D1/L/M/Q子目录已建
- [ ] 证据分类编号未跳号

### 归档检查清单

- [ ] 工作成果完整
- [ ] 案件文件夹已移至 Archives/{YYYY}/{MM}/

- [ ] ID保留不变

## 10. 版本管理

- SOP文件、元规则文件更新时，旧版本移入 `05-归档/`
- 归档文件名格式：`{原文件名}-{版本号}-旧版.md`
- 版本号遵循语义化版本（major.minor.patch）

## 11. 与v1.0的变更摘要

| 变更项 | v1.0 | v2.0 |
|--------|------|------|
| 桌面层数 | 三层（Inbox/Active/Archives） | 四层（+LIBRARY） |
| 扫描频率 | 每日凌晨1次 | 每日10:00 + 22:30 共2次 |
| Inbox迁移 | 纯人工触发 | AI主动处理（含"1"标记） |
| 案件触发 | 无 | 文件夹名含"1" |
| MD指令 | 无 | 有MD按SOP走，无MD标"待处理" |
| Active归档 | 手动 | 超4天自动归档 |
| Archives分类 | 未定义 | 按年/月时间归档 |
| 非案件材料 | 全部入Inbox | 直接入LIBRARY |
