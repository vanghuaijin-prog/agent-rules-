# 证据整理与合订本制作 SOP v2

> **适用范围**：手上有一堆零散的诉讼证据材料（图片、PDF、聊天记录截图、Excel、音视频等），需要从"乱文件夹"状态整理到编号清晰、映射完善、按证明组合并的法院级证据合订本。
>
> **本SOP是「诉讼案件分析SOP」Phase 1（事实与证据）的详细展开版，可以独立使用，也可以嵌入全流程。**
>
> **读者**：律师、AI助手、法律助理。如果你是AI，请严格按本SOP的步骤执行，不要跳过任何"看起来显然"的步骤。

---

## 一、在整体诉讼流程中的位置

完整的诉讼立案准备分为六个阶段：

| Phase | 名称 | 核心产出 |
|-------|------|----------|
| **Phase 1** | **事实与证据** | **证据分类编号、映射表、PDF合订本 ← 本SOP覆盖这里** |
| Phase 2 | 法律研究 | 案例检索 + 法条分析备忘录（内部文件） |
| Phase 3 | 金额计算 | 诉讼请求计算明细表 |
| Phase 4 | 起诉状 | 民事起诉状.docx |
| Phase 5 | 证据清单 | 六列法院格式表格（从映射表自动生成） |
| Phase 6 | 程序性文件 | 身份证、委托书、工商登记信息等 |

本SOP只覆盖Phase 1，但它产出的映射表和合订本是后续所有阶段的基石。

## 二、核心设计原则

> 这些原则贯穿整个SOP。如果你（AI）在执行过程中遇到模糊地带，回到这里找方向。

### 2.1 先讨论再动手

在关键决策点（分类方案、证明组设计、跨组处理策略），**先向用户展示方案、等用户确认后再执行**。不要自作主张直接干。原因：分类错了要返工，证明组设计错了要重做合订本，成本远高于多问一句。

典型卡点：
- 初次分类完成后，展示分类结果让用户确认
- 证明组字母定义完成后，展示让用户确认
- 发现证据金额与诉求金额不一致时，立刻提醒，不默默改

### 2.2 ID是锚，页码是座位

- **证据ID**（如 P1-02-01）= 身份证号码，永久不变。新增证据不改已有ID，只追加新ID。
- **页码**（如 第5页）= 电影院的座位号，每次重建合订本时自动重算。

所以：新增证据 → 追加ID → 重跑合并脚本 → 页码自动更新。不需要手动改任何已有编号。

### 2.3 幂等操作

合订本合并脚本是**每次从头生成**，不是增量修改。任何修改（加证据、改DPI、改排序）只需要改输入参数然后重跑脚本。不需要理解"上次跑到哪了"。

### 2.4 映射表 ≠ 证据清单

这是两个完全不同的东西，不要搞混：

| 维度 | 映射表 | 证据清单 |
|------|--------|----------|
| 用途 | 内部工作底稿（你+AI用） | 提交给法院的正式文件 |
| 格式 | Excel/Markdown，七列 | Word/PDF，六列表格 |
| 内容 | 包含备注、证明组代号等内部信息 | 只有法院需要的标准字段 |
| 生成时机 | 分类编号完成后立刻做 | PDF合订本定稿后最后做 |

### 2.5 文件夹版本管理

每次重大操作前，**先复制整个文件夹**，命名带版本号。这样操作出错可以回退。

```bash
cp -r 试验品-v1 试验品-v2    # 分类前备份
cp -r 试验品-v1 试验品-v2 试验品-v3  # 合并PDF前备份
```

命名习惯：`{案件名}-第{N}步` 或 `{案件名}-v{N}`。

**材料包文件夹命名（推荐）**：

```
{YYYYMMDD}{原告}诉{被告}{案由}-材料包-v{N}
```

示例：`20260624陈玉欣诉周万略健康权纠纷案-材料包-v1`

材料包里**只放组织好的结构**（P1/D1/L/M/Q），原始材料文件夹留在外面作为存档，不要混在一起。这样整理完就是干净的交付物，不会被原始 hash 文件和乱文件夹名污染。

## 三、四级证据编号体系

### 3.1 编号格式

```
[角色代码]-[大类编号]-[条目编号][-子项字母]
```

示例：`P1-02-03-a` = 原告1 → 财务转账凭证 → 第3份 → 子项a

### 3.2 角色代码

| 代码 | 含义 | 说明 |
|------|------|------|
| P1, P2, P3... | Plaintiff（原告） | 多个原告时递增编号。单原告就是 P1 |
| D1, D2, D3... | Defendant（被告） | 多个被告时递增编号 |
| T1, T2... | Third party（第三人） | 如有第三人 |
| L | Lawyer（律师工作成果） | AI产出的分析、清单、合订本等 |
| M | Mapping（映射表） | 证据映射表本身 |
| **Q** | **Queue（待补充队列）** | 临时收纳区，见§3.6 |

### 3.3 大类编号

**01-当事人身份材料**是通用类，几乎必有，固定占用01编号，可以先建。

**其他大类从02开始按实际材料顺序编号。** 不是照搬下面的参考表编号——参考表只列类别名称和典型内容，编号是示例而非模板。你的案子只有身份材料、客户说明、对方书面确认三类证据，那就编01、02、03，不要跳到07、08。空文件夹暗示"这里应该有东西"，容易误导，没有的类别不建。

以下是常见大类的参考（类别名称参考，编号按实际顺序）：

| 类别名称 | 典型内容 |
|----------|----------|
| 当事人身份材料（固定01） | 身份证、委托书、营业执照 |
| 财务转账凭证 | 银行回执、转账记录、流水 |
| 收据押金凭证 | 收据、押金条 |
| 发票 | 增值税发票、普通发票 |
| 沟通记录 | 微信聊天截图、语音、视频录屏 |
| 销售/业务数据 | 销售报表、提成计算表、合同 |
| 客户/当事人说明 | 当事人出具的情况说明或和当事人的沟通记录 |
| 对方书面确认 | 对方出具的欠条、说明、确认函 |

> **⚠️ 分类的核心判断标准**：这份材料是"对话过程中的截取"（归沟通记录），还是"一方正式出具的书面文件"（归对方书面确认）？后者证明力远强于前者。典型案例：一张"说明"截图看起来像聊天记录，但它是被告分公司正式出具的、有签名有日期有金额确认的文件，应该归对方书面确认而非沟通记录。

### 3.4 文件命名规范

#### 命名格式

```
[证据ID]-[可读描述].[扩展名]
```

#### 命名决策树

按顺序回答以下问题，把答案拼进文件名：

1. **这份文件的性质是什么？** → 身份证 / 银行回执 / 收据 / 说明 / 微信聊天截图 / ...
2. **涉及哪些主体？** → 如有资金流向，用 `→` 表示（如"仁康→正和"）
3. **涉及多少金额？** → 精确到分，不带千位分隔符，不加"元"字（如 21185.69）
4. **需要日期区分吗？** → 仅在同名文件多份、需要区分时才加日期

**填空模板**：`[性质]-[主体1]→[主体2]-[金额].[扩展名]`

示例：
- `P1-01-01-辛丽伟身份证.jpg` — 性质=身份证，主体=辛丽伟
- `P1-02-01-建设银行回执-仁康→正和-21185.69.jpg` — 性质=银行回执，主体=仁康→正和，金额=21185.69
- `P1-08-01-说明-欠款28338.66.png` — 性质=说明，金额=28338.66

> **⚠️ 原始hash文件名（如 `09c739367fde9fa54d824853daf13312.jpg`）必须在整理阶段改掉。** 以后查找全靠文件名，hash名 = 没名字。

### 3.5 子项字母（-a, -b, -c）的使用规则

子项字母表示"同一份证据的多个组成部分"。使用条件：

| 场景 | 编法 | 理由 |
|------|------|------|
| 身份证正反面两张图 | P1-01-01-a（正面）、P1-01-01-b（反面） | 同一份证件的两个面 |
| 同一次转账的四张截图（不同角度） | P1-02-01-a ~ P1-02-01-d | 同一笔交易的多个视角 |
| 两次不同的转账 | P1-02-01 和 P1-02-02 | 两笔独立交易，各是各 |
| 同一份收据的两张照片 | P1-03-01-a 和 P1-03-01-b | 同一份东西的两个角度 |

**判断标准**：如果它们在法庭上会被当作"一份证据的X页"，就用子项；如果会被当作"X份独立证据"，就各编各的号。

### 3.6 Q-Queue（待补充队列）

在根目录建一个 `Q-Queue/` 文件夹。新材料先丢进 Q-Queue，**不急着编号**。等收集齐了、确认要用了，再走正常流程：分类 → 编号 → 命名 → 移入正式目录 → 更新映射表。

> **为什么不在收到时立刻编号？** 很多补充材料最后可能用不上（重复的、模糊的、和案件无关的）。提前编号会导致编号跳号或者映射表反复修改。Q-Queue 是缓冲区，进了正式序列才算数。

## 四、文件夹结构搭建与文件处理

### 4.1 根目录结构

```
{案件名}/
├── P1-原告({姓名})/
│   ├── 01-当事人身份材料/
│   ├── 02-财务转账凭证/
│   ├── 03-收据押金凭证/
│   ├── ...（根据实际材料动态创建）
├── D1-被告({公司名})/
├── T1-第三人/（如有）
├── L-律师工作成果/
│   ├── 01-文书/（证据清单.docx 等）
│   └── 02-证据合并输出/（合订本PDF放这里）
├── M-证据映射表/
│   └── M-00-证据映射表.xlsx
└── Q-Queue/（待补充材料缓冲区）
```

### 4.2 读取文件的方法

**在分类之前，你必须先"看到"每份文件的内容。** 不同类型的文件，读取方法不同：

| 文件类型 | 读取方法 | 你能看到什么 |
|----------|----------|------------|
| JPG / PNG 图片 | Read工具直接打开 | 图片内容 |
| PDF（文本型） | Read工具 | 文字内容 |
| PDF（扫描件） | Read工具查看图片 | 扫描后的图片 |
| Word (.docx) | Read工具 | 文字内容 |
| Excel (.xlsx) | Read工具或pandas读取 | 表格数据 |
| 视频 (.mp4) | macOS: `qlmanage -t -s 1280 -o /tmp/ {视频路径}` 抽取缩略图 | 一帧画面截图 |
| 音频 (.mp3等) | 无法直接"听"。提示用户转写或手动记录要点 | 无 |

> **⚠️ 视频处理**：以后尽量让当事人提供截图而非视频。视频的信息密度远低于截图+文字说明。如果已经有视频，用上面的方法提取关键帧截图。如果视频内容复杂（多页文档的录屏），可能需要抽多帧：`ffmpeg -i video.mp4 -vf "fps=1" /tmp/frames/%03d.jpg`（每秒一帧）。

### 4.3 分类决策树

**逐份文件走以下决策树，确定它应该放在哪个大类下：**

```
拿到一份文件
    │
    ├─ 问题1：谁出具的？
    │   ├─ 自己方（原告/委托人）→ 继续
    │   ├─ 对方（被告）→ 继续
    │   └─ 第三方（银行/政府/平台）→ 继续
    │
    ├─ 问题2：是正式文件还是对话截取？
    │   ├─ 正式出具的文件（有签名/公章/日期/金额确认）
    │   │   ├─ 对方出具的 → 【08-对方书面确认】
    │   │   └─ 自己方出具的 → 【07-客户/当事人说明】
    │   └─ 对话/沟通过程的截取（聊天截图/语音/视频录屏）
    │       └─ 【05-沟通记录】
    │
    ├─ 问题3：内容属于什么业务环节？
    │   ├─ 身份信息（身份证/营业执照/委托书）→ 【01-当事人身份材料】
    │   ├─ 资金流转（银行回执/转账记录/流水）→ 【02-财务转账凭证】
    │   ├─ 收款确认（收据/押金条）→ 【03-收据押金凭证】
    │   ├─ 发票 → 【04-发票】
    │   └─ 业务数据（销售表/提成表/合同）→ 【06-销售/业务数据】
    │
    └─ 无法判断 → 放入 Q-Queue，标记"待确认"，问用户
```

> **⚠️ 做完初步分类后，必须逐份复核。** 对每一份文件问自己："这份材料的法律性质是什么？"发现不对劲就建新类、移文件、更新映射表。

### 4.4 操作步骤（完整流程）

1. **复制文件夹**：`cp -r {原始文件夹} {原始文件夹}-第二步`（版本管理。最终材料包命名见§2.5，如 `20260624陈玉欣诉周万略健康权纠纷案-材料包-v1`）
2. **创建根目录结构**：建 P1/D1/L/M/Q 等文件夹（只建01-身份材料，其他等材料决定后再建）
3. **逐份读取文件**：按§4.2的方法读取每份文件内容
4. **逐份分类**：按§4.3的决策树判断类别，按需创建大类子文件夹
5. **逐份编号+重命名**：按§3.4的命名规范重命名
6. **移入对应文件夹**
7. **检查根目录已清空**（除文件夹外无散落文件）
8. **展示分类结果给用户确认**（先讨论再动手原则）
9. **分类复核**：逐份检查是否有分类不当（特别关注"正式文件 vs 沟通记录"）

## 五、证据映射表制作

### 5.1 映射表的定位

映射表是整个体系的"中枢神经"。它用一张表格把"按文书类型分类的物理文件"和"按证明目的分类的逻辑分组"连接起来。同一份证据可以同时出现在多个证明组中。

**工作阶段用Markdown**（纯文本，AI读写零成本，不依赖任何库）。**交付阶段转xlsx**（律师审阅、打印用）。

### 5.2 七列结构

| 列名 | 说明 | 示例 |
|------|------|------|
| 证据ID | 四级编号 | P1-02-01 |
| 文件名 | 可读描述（不含ID前缀，ID列已有） | 建设银行回执-仁康→正和-21185.69 |
| 来源 | 谁提供的 | 原告 |
| 待证事实（简） | 一句话简写，人快速扫用 | 证明货款流向 |
| 证明对象（完整版） | 完整的法院级事实句 | 证明2023年X月X日，仁康医药通过建设银行向通化正和转账21185.69元，系销售回款 |
| 证明组 | 属于哪些证明组 | A, C |
| 备注 | 简练的补充说明 | 仁康医药→通化正和 |

### 5.3 证明对象（完整版）写法

**硬规则**：完整事实句，包含时间 + 主体 + 金额/行为 + 法律意义。

| 好的写法 | 不好的写法 |
|----------|----------|
| 证明2023年4月至2024年1月间，原告通过仁康医药向被告通化正和共计转账13503.54元，系内蒙九州通药品销售回款。 | 转账记录 ❌（太简） |
| | 银行回执 ❌（没说证明什么） |

### 5.4 证明组设计方法

**不要凭空编证明组。从诉讼请求倒推：**

```
步骤1：写下诉讼请求
   "请求判令被告返还提成款17338.66元 + 押金10000元"

步骤2：拆解每个请求的构成要件
   提成款17338.66元 →
     要件a：双方存在劳动/代理关系（有权拿提成）
     要件b：提成金额确为17338.66元（数额确定）
     要件c：被告未支付该提成（未履行义务）
   押金10000元 →
     要件d：原告已交付押金10000元（已交付）
     要件e：被告未返还押金（未履行义务）
   通用 →
     要件f：原告积极主张权利（未放弃/诉讼时效未过）

步骤3：每个要件 = 一个证明组
   A = 劳动/代理关系存在       ← 要件a
   B = 押金已交付10000元       ← 要件d
   C = 提成款金额17338.66元    ← 要件b
   D = 被告未返还款项          ← 要件c+e
   E = 原告积极主张权利        ← 要件f
```

> **⚠️ 证明组字母的含义必须在映射表主表里直接可见。** 数据区下方空一行，直接贴索引表。不要只放在第二个sheet里——拉到表底下一眼就看到最方便。

映射表里的索引格式：

| 证明组 | 待证事实 |
|--------|----------|
| **A** | 劳动/代理关系存在 |
| **B** | 押金已交付10000元 |
| **C** | 提成款金额17338.66元 |
| **D** | 被告未返还款项 |
| **E** | 原告积极主张权利 |

### 5.5 跨组引用规则

同一份证据可能同时证明多个事实。处理方式：

**在映射表中**：证明组列写所有相关字母（如 `B, C, D`）。

**在分组PDF合订本中**：

| 规则 | 说明 |
|------|------|
| 分组PDF | 允许同一证据出现在多个组的PDF中（方便单独查阅某一组） |
| 合订本 | **默认不重复**——只放在主要证明组的PDF里，次要组在证据清单中写"详见组X第X号证据" |

> **如何判断"主要证明组"？** 问：这份证据最核心、最直接证明的是哪个事实？那就是它的主要组。比如"说明"文件最核心证明的是"被告确认欠款金额"（C），押金交付（B）和未返还（D）是附带证明。

### 5.6 三处同步更新规则

**任何一次证据变动（移类、删证据、加证据），映射表必须同时更新三处：**

1. **主表**：对应行的证据ID/文件名/备注
2. **证明组包含列表**：如果证据换了证明组，更新相关组的包含列表
3. **关联备注**：如果某证据被替代，在替代它的证据备注中注明"替代XX"，在原证据备注中注明"已由XX替代"

> **⚠️ 漏改任一处 = 映射表不一致。** 不一致的映射表比没有映射表更危险——你会基于错误信息做决策。

### 5.7 关键证据标记

对案件有决定性作用的证据，用**红色加粗 + ★** 标记。

典型关键证据：对方出具的书面确认文件（欠条、说明、对账单），因为它同时证明多个待证事实且证明力极强。

> **xlsx 实现**：在 Excel 中，选中对应行的文件名单元格 → 字体颜色设为红色(RGB 255,0,0) → 加粗 → 文件名前手动加★字符（如 `★P1-08-01-说明-欠款28338.66`）。Markdown 工作阶段用 `**★红色**` 标记即可。

### 5.8 备注风格

备注要**简练精当**，写人话：

| 场景 | 好的备注 | 不好的备注 |
|------|----------|------------|
| 转账 | 仁康医药→通化正和 | 该证据显示仁康医药通过建设银行向通化正和进行了转账 |
| 重复 | 与01-03重复 | 该证据与编号为P1-01-03的证据存在内容上的重叠 |
| 需处理 | 需查看内容后归类 | 待进一步审查以确定其分类归属 |
| 被替代 | 已由08-01替代 | （同上啰嗦版） |

## 六、文件格式处理规则

不是所有文件都能直接合并进 PDF：

| 文件类型 | 处理方式 | 谁来做 |
|----------|----------|--------|
| JPG / PNG 图片 | 直接合并 | AI |
| PDF（A4尺寸） | 直接合并 | AI |
| PDF（非A4尺寸） | 渲染为图片→放A4页→合并 | AI |
| Excel (.xlsx) | **截图保存为图片** → 合并 | **用户手动** |
| Word (.docx) | **导出为 PDF** → 合并 | **用户手动** |
| 视频 (.mp4) | **无法合并。** 截屏关键画面为图片→合并，或刻盘/U盘单独提交 | **用户手动** |
| 音频 (.mp3等) | **转文字稿→Word→PDF→合并** | **用户手动** |

> **⚠️ 遇到非PDF友好格式时，提示用户手动处理，不要硬转。** AI把Excel直接转PDF排版会乱，把Word硬转PDF字体可能不对。让用户自己截图/导出更靠谱。

## 七、PDF合并与页码标注

### 7.1 合并策略

按证明组分别合并，最后拼合订本：

```
L-律师工作成果/02-证据合并输出/
├── 证据组A-{待证事实名}.pdf
├── 证据组B-{待证事实名}.pdf
├── ...
└── 证据合订本-{案件名}.pdf
```

### 7.2 组内证据排序

**按叙事逻辑排序，不是按编号排序。** 法官读证据是顺着故事线读的。

典型叙事顺序：
1. 建立关系（委托书、合同）
2. 履行行为（转账、交货、提供服务）
3. 违约事实（未付款、未返还）
4. 金额确认（对账单、说明）
5. 权利主张（催款记录、律师函）

### 7.3 双层页码标注

每一页叠加两个信息：

| 位置 | 内容 | 字号 | 颜色 |
|------|------|------|------|
| 底部居中 | `- {页码} -` | 11pt | 深灰(80,80,80) |
| 右上角 | 证据ID | 9pt | 浅灰(140,140,140) |

> **⚠️ 页码是图层叠加，不修改原文件。** 不管底下是截图、扫描件还是文字PDF，页码都能加。

### 7.4 依赖库安装

合并脚本需要三个Python库。执行前先确保已安装：

```bash
pip install Pillow pypdf pypdfium2 --quiet
```

如果某个库装不上：
- Pillow 装不上 → 无法处理图片，只能用已有PDF
- pypdf 装不上 → 无法合并PDF，需要用户手动合并
- pypdfium2 装不上 → 无法渲染PDF页面为图片，A4和非A4的PDF都无法处理（见§7.6易错点1）

> **注**：脚本对 A4 PDF 使用 300 DPI 渲染（高于默认 150 DPI），以最大限度减少光栅化带来的质量损失。DPI 值可通过修改脚本顶部的 `DPI` 常量统一调整，A4 渲染 DPI 硬编码为 300。

### 7.5 完整合并脚本

以下脚本可直接使用。修改 `BASE`、`P1`、`OUT` 路径和 `groups` 字典即可适配不同案件。

```python
import os, io
from PIL import Image, ImageDraw, ImageFont
import pypdf
import pypdfium2 as pdfium

# ===== 修改这些参数 =====
BASE = os.path.expanduser("~/Desktop/{案件文件夹}")
P1 = f"{BASE}/P1-原告({姓名})"
OUT = f"{BASE}/L-律师工作成果/02-证据合并输出"
os.makedirs(OUT, exist_ok=True)

FONT_PATH = "/System/Library/Fonts/STHeiti Medium.ttc"  # macOS中文字体
DPI = 150          # 清晰度：150=标准，200=更清晰，300=印刷级
A4_W = int(595 * DPI / 72)   # A4宽（像素）
A4_H = int(842 * DPI / 72)   # A4高（像素）
MARGIN = int(40 * DPI / 72)  # 页边距
# ========================

def img_to_pdf_page(img_path, evidence_id, page_num):
    """图片 → A4尺寸PDF页，叠加双层页码。"""
    img = Image.open(img_path).convert("RGB")
    
    avail_w = A4_W - 2 * MARGIN
    avail_h = A4_H - 2 * MARGIN - int(80 * DPI / 72)  # 底部留页码空间
    
    scale = min(avail_w / img.width, avail_h / img.height)
    new_w, new_h = int(img.width * scale), int(img.height * scale)
    img_resized = img.resize((new_w, new_h), Image.LANCZOS)
    
    canvas = Image.new("RGB", (A4_W, A4_H), (255, 255, 255))
    x_off = (A4_W - new_w) // 2
    y_off = (A4_H - int(80 * DPI / 72) - new_h) // 2 + int(30 * DPI / 72)
    canvas.paste(img_resized, (x_off, y_off))
    
    draw = ImageDraw.Draw(canvas)
    font_num = ImageFont.truetype(FONT_PATH, int(11 * DPI / 72))
    font_id = ImageFont.truetype(FONT_PATH, int(9 * DPI / 72))
    
    # 底部居中：连续页码
    txt = f"- {page_num} -"
    bbox = draw.textbbox((0,0), txt, font=font_num)
    tw = bbox[2] - bbox[0]
    draw.text(((A4_W - tw) // 2, A4_H - int(35 * DPI / 72)),
              txt, fill=(80,80,80), font=font_num)
    
    # 右上角：证据ID
    draw.text((A4_W - MARGIN - int(80 * DPI / 72), int(12 * DPI / 72)),
              evidence_id, fill=(140,140,140), font=font_id)
    
    buf = io.BytesIO()
    canvas.save(buf, format="PDF", resolution=DPI)
    buf.seek(0)
    return buf

def pdf_to_images(pdf_path):
    """非A4尺寸的PDF → PIL图片列表。跳过极小页面（<200px）。"""
    doc = pdfium.PdfDocument(pdf_path)
    images = []
    for i in range(len(doc)):
        page = doc[i]
        bitmap = page.render(scale=DPI/72)
        pil_img = bitmap.to_pil()
        if pil_img.width < 200 or pil_img.height < 200:
            print(f"    skip tiny page {i+1}: {pil_img.size}")
            continue
        images.append(pil_img)
    return images

def is_a4_page(pdf_path, tolerance=5):
    """检测PDF首页是否为A4尺寸（595×842 points），容差tolerance点。"""
    reader = pypdf.PdfReader(pdf_path)
    if not reader.pages:
        return False
    box = reader.pages[0].mediabox
    w, h = float(box.width), float(box.height)
    return (abs(w - 595.28) < tolerance and abs(h - 841.89) < tolerance) or \
           (abs(h - 595.28) < tolerance and abs(w - 841.89) < tolerance)

# ===== 定义证明组及组内证据排序（按叙事逻辑） =====
# 每项: (证据ID, 文件完整路径, 类型"img"或"pdf")
groups = {
    "A-{待证事实名}": [
        ("P1-01-01", f"{P1}/01-.../01-01-xxx.jpg", "img"),
        # ... 按叙事顺序排列
    ],
    "B-{待证事实名}": [
        # ...
    ],
    # ... 更多组
}
# ================================================

page_num = 1
final_writer = pypdf.PdfWriter()
group_info = {}

for group_name, items in groups.items():
    group_writer = pypdf.PdfWriter()
    start_page = page_num
    print(f"\n=== 组 {group_name} ===")
    
    for eid, fpath, ftype in items:
        if not os.path.exists(fpath):
            print(f"  SKIP {eid}: file not found at {fpath}")
            continue
        
        if ftype == "img":
            buf = img_to_pdf_page(fpath, eid, page_num)
            reader = pypdf.PdfReader(buf)
            for pg in reader.pages:
                group_writer.add_page(pg)
                final_writer.add_page(pg)
            print(f"  p{page_num} {eid}")
            page_num += 1
            
        elif ftype == "pdf":
            a4 = is_a4_page(fpath)
            if a4:
                # A4 PDF：高质量渲染(300DPI)，减少不必要的光栅化损失
                doc = pdfium.PdfDocument(fpath)
                images = []
                for i in range(len(doc)):
                    page = doc[i]
                    bitmap = page.render(scale=300/72)  # 高于默认DPI，保证质量
                    pil_img = bitmap.to_pil()
                    if pil_img.width < 200 or pil_img.height < 200:
                        continue
                    images.append(pil_img)
            else:
                # 非A4 PDF：标准光栅化路径（避免直接合并出空白页）
                images = pdf_to_images(fpath)
            for pi, pil_img in enumerate(images):
                tmp_path = "/tmp/_tmp_pdf_page.png"
                pil_img.save(tmp_path)
                buf = img_to_pdf_page(tmp_path, eid, page_num)
                reader = pypdf.PdfReader(buf)
                for pg in reader.pages:
                    group_writer.add_page(pg)
                    final_writer.add_page(pg)
                dpi_used = 300 if a4 else DPI
                print(f"  p{page_num} {eid} (pdf page {pi+1}, {'A4' if a4 else 'non-A4'} {dpi_used}dpi)")
                page_num += 1
    
    end_page = page_num - 1
    count = len(group_writer.pages)
    group_info[group_name] = (start_page, end_page, count)
    
    group_path = f"{OUT}/证据组{group_name}.pdf"
    with open(group_path, "wb") as f:
        group_writer.write(f)
    print(f"  -> {count} pages (p{start_page}-p{end_page})")

# 保存合订本
final_path = f"{OUT}/证据合订本-{案件名}.pdf"
with open(final_path, "wb") as f:
    final_writer.write(f)

print(f"\n{'='*50}")
print(f"总页数: {len(final_writer.pages)} ({DPI} DPI)")
for gn, (s, e, c) in group_info.items():
    print(f"  组{gn}: p{s}-p{e} ({c}页)")

# 验证页面尺寸一致性
expected_w = A4_W * 72 / DPI
expected_h = A4_H * 72 / DPI
for i, pg in enumerate(final_writer.pages):
    w, h = float(pg.mediabox.width), float(pg.mediabox.height)
    if abs(w - expected_w) > 5 or abs(h - expected_h) > 5:
        print(f"\n⚠️ 第{i+1}页尺寸异常: {w:.0f}x{h:.0f} (期望 {expected_w:.0f}x{expected_h:.0f})")
        break
else:
    print("\n✅ 全部页面尺寸一致")
```

### 7.6 易错点

> **⚠️ 易错点1：非A4 PDF直接合并会出空白页。** `pypdf.merge_transformed_page()` 对非标准尺寸的PDF页（如Excel导出的699×768像素页）会丢失内容。**必须**先用 pypdfium2 渲染为图片，再按图片处理。脚本中的 `pdf_to_images()` 函数已处理此问题。
>
> **⚠️ 易错点2：极小页面要跳过。** 有些PDF的第2页只有122×74像素（空白页脚碎片），不跳过会产生空白页。脚本中 `pdf_to_images()` 设置了 200px 阈值。
>
> **⚠️ 易错点3：中文引号在 Python heredoc 中会冲突。** 如果用 bash heredoc 写 Python 脚本，中文引号 `""` 可能和 Python 字符串分隔符冲突。解决方案：用 unicode 转义 `\u201c\u201d`，或改用单引号，或把脚本写成 .py 文件再执行。
>
> **⚠️ 易错点4：中文字体路径。** macOS 用 `/System/Library/Fonts/STHeiti Medium.ttc`。如果路径不对，Pillow 会报错找不到字体。Linux 上需要改成对应路径（如 `/usr/share/fonts/truetype/wqy/wqy-zenhei.ttc`）。

### 7.7 DPI与清晰度

| DPI | 效果 | 文件大小（21页） |
|-----|------|----------------|
| 150 | 标准，够法庭用 | 3-5MB |
| 200 | 更清晰 | 5-8MB |
| 300 | 印刷级 | 10-15MB |

只需改脚本里的 `DPI` 常量，重跑即可。

## 八、证据清单生成

### 8.1 生成时机

**PDF合订本定稿后再生成，不做草稿再回填。**

理由：页码取决于最终PDF排版，先出清单留空白页码 = 后面逐项回填 = 工作量翻倍。映射表里已有所有数据字段，从映射表自动生成六列表格是一次到位的。

唯一例外：团队需要先审阅文字表述时，可先出页码栏写"见合订本第__页"的草稿。

### 8.2 法院六列格式

| 编号 | 证据名称 | 份数 | 页数 | 原件或复印件 | 证明对象 |
|------|----------|------|------|------------|----------|

**"原件或复印件"判断标准**：

| 情况 | 填写 |
|------|------|
| 有红章/公章的纸质原件 | 原件 |
| 身份证/执照的照片 | 复印件 |
| 银行电子回执（彩色打印） | 复印件 |
| 聊天截图 | 复印件 |
| 对方出具的书面文件照片 | 复印件 |

> 实务中大多数证据都是复印件（照片/截图/打印件），除非你手上有盖章原件。

### 8.3 跨组证据的处理

在次要证明组的证据清单行中，"页数"列写"详见组X第X号证据"，不需要重复计算页数。

**示例**：证据 P1-08-01（说明-欠款28338.66）主要属于组C，同时关联组B和组D。则：
- 组C的证据清单正常写页数（如"1页"）
- 组B和组D的证据清单中，该证据的"页数"列写：`详见组C第3号证据`

## 九、常见陷阱与应对

### 9.1 证据金额与诉求金额不一致

**典型场景**：整理证据过程中发现一份"说明"文件，确认的总金额比起诉状写的多（比如多了退货保证金1000元）。

**应对**：**这是强制检查步骤**——整理完所有证据后，加总证据中出现的所有金额，与起诉状诉求金额逐项比对。有差异立刻提醒律师，不默默改。起诉状金额变更可能影响管辖、诉讼费计算、保全金额。

### 9.2 分类错误

**典型场景**：把被告正式出具的书面文件放进了"沟通记录"。

**应对**：见§3.3 的判断标准和§4.3 分类决策树。发现分错后立即建新类、移文件、三处同步更新映射表（§5.6）。

### 9.3 编号跳号

**典型场景**：删除了一份证据后，编号不连续。

**应对**：允许跳号，不改号。编号是身份标识（§2.2），不是序号。改一个号需要更新映射表、证据清单、合订本页码，牵一发动全身。删除的证据在映射表中标记"已删除"即可，编号保留空缺。

### 9.4 新证据插入

**应对**：
1. 新材料先进 Q-Queue（§3.6）
2. 确认要用后，分配新编号（在对应大类下递增）
3. 命名、移入正式目录
4. 三处同步更新映射表（§5.6）
5. 重新跑合并脚本（§7.5），页码自动重算（§2.3）

### 9.5 非A4尺寸的PDF页

**典型场景**：从Excel导出的销售数据PDF，页面尺寸不是A4。

**应对**：脚本中的 `pdf_to_images()` 函数已自动处理——用 pypdfium2 渲染为图片后按图片流程处理。详见§7.6 易错点1。
