# revision-strategy

---

## 一、总原则

### 1. 最小必要修改原则

- 目标不是“改得越多越好”，而是“用最小必要动作把风险处理到可签、可谈、可执行”。
- 能通过局部删减、局部补入解决的问题，不要整段重写。
- 能通过直接修订解决的确定性问题，不要只留批注把工作再抛回给客户。

### 2. 确定性问题优先直接改

- 错别字、主体名称错误、金额/日期前后不一致、交叉引用错误、编号错误、术语不统一等，原则上直接修订。
- 明显违法、明显失衡、明显不可执行的表述，原则上直接给出可落地修改。

### 3. 商务取舍和事实缺口优先批注

- 涉及价格、比例、期限、范围、商业让步、资源投入等取舍，若没有明确授权边界，优先批注提示，不强行代替客户做决定。
- 涉及项目名称、金额、日期、地址、附件清单、履行节点等留空信息，原则上不代填，只做批注提示。

### 4. 不为“精简”而随意删减

- 法条引用、定义性陈述、解释性前言，只要准确、不冲突、不误导，通常无需仅因“写得多”而删减。
- 冗长本身不是风险；错误、冲突、失衡、不可执行才是风险。

### 5. 修订版要让客户看得懂

- 修订模式下，应尽量让 Word 审阅视图清楚呈现“删了什么、补了什么、为什么改”。
- 能展示为局部删除或局部插入的，不要把整段都替换成新段落。
- 整段重写只作为最后手段，而不是默认动作。
- 也不要为了“显得改得很多”而把一条款拆成过多单字级、标点级碎修订；若同一条款存在大量细碎差异，应优先合并成更少、更成组的修订块。

## 二、动作选择顺序

## 1. 先判断：这是不是确定性问题？

- 是：优先直接修订。
- 否：进入下一步，判断是否属于商务取舍或事实待补。

## 2. 再判断：这是不是客户自己必须确认或补录的信息？

- 是：优先批注。
- 否：进入下一步，判断能否局部删减或局部补入。

## 3. 能否用更小的动作解决？

- 能仅删掉一小段风险性表述：优先 `delete`。
- 能仅补一句触发条件、责任边界、通知机制、例外条款：优先 `insert`。
- 能仅替换局部短语、数字、主语、期限、机构名称、争议解决条款中的关键元素：优先局部 `replace`。
- 只有当原条款整体逻辑失效，局部动作已无法修复时，才使用整条 `replace`。

## 4. 最后判断：是否只进意见书或保留批注，而不直接改文？

- 纯优化型、程序型、说明型问题，可只进入审查意见书，不必在 Word 页面留痕。
- 涉及客户立场、谈判空间、商业选择、事实未明、附件缺失时，优先保留批注。
- 批注中应说明：问题是什么、风险在哪里、客户需要补什么，必要时附备选措辞。

## 5. 不要把“识别到风险”等同于“必须改正文”

- 风险识别应尽量完整，但正文改动应当克制。
- 不是每一个风险点都必须进入修订版或批注版。
- 低必要性、低确定性、纯优化建议，可以只写进审查意见书。

## 三、典型情形与默认动作

| 情形 | 默认动作 | 处理说明 |
|:---|:---:|:---|
| 纯优化型、程序型、说明型建议 | `report-only` | 保留在审查意见书中，不必在 Word 页面留下修订或批注痕迹。 |
| 合同留空、占位符、待填字段 | `comment` | 不代填金额、日期、地址、附件名称、项目名称、履行节点等；提醒客户补充即可。 |
| 主体名称、日期、金额、编号、交叉引用前后不一致 | `replace` | 属于确定性错误，直接修正。 |
| 重复表述、明显赘余、冲突性句子 | `delete` | 能删掉风险句而不破坏主条款结构的，优先局部删除。 |
| 缺少通知方式、送达地址、争议机构、责任上限、解除后果等关键补位 | `insert` 或局部 `replace` | 以最小补入为先；仅在原条款结构明显失效时整条重写。 |
| 单方免责、无限责任、任意解除、过重违约金 | `delete` + `insert` / 局部 `replace` 优先 | 能通过删减单方倾斜表述并补入边界的，不要整段重写。 |
| 条款整体逻辑错误，局部改动已无法修复 | 整条 `replace` | 例如付款机制、验收机制、交付机制整条失衡或自相矛盾。 |
| 商业数字、比例、期限存在谈判空间 | `comment` 为主 | 若客户未明确授权，不替其直接改价、改比例、改期限。 |
| 法条引用准确但较长 | 通常不处理 / 必要时 `comment` | 不因“法条多”而机械删减；只有错误、过时、冲突或误导时才修改。 |
| 格式、编号、标点、称谓统一 | `replace` | 直接修正，通常不必保留批注。 |
| 附件缺失、事实背景不足、条件不明 | `comment` | 说明缺什么、影响什么、需谁补。 |

## 四、几个高频边界

### 1. 留空项

- 像“签订日期：____”“金额：____”“附件：____”“交付地址：____”这类内容，原则上不直接代填。
- 这类问题的价值不在于“帮客户随便写上去”，而在于提醒其补齐并意识到不补的后果。
- 默认处理：保留原文 + 批注提示。
- 即使上游审查计划误把这类问题标成 `replace`，执行阶段也应回退为批注，而不是把待填信息直接落进正文。

### 2. 法条引用

- 如果法条引用准确、适配当前交易、且没有与正文冲突，不因篇幅较长而主动删减。
- 如果法条引用错误、过时、与最新法制不符，或被当作免责/排除责任的错误依据，应当直接修正或删除错误部分。
- 默认处理：准确则保留；错误则直接改。

### 3. 违约责任和解除条款

- 这类条款最容易“整段重写过度”。
- 先看能否通过删掉单方倾斜句、补入责任边界、补入通知期限来完成修复。
- 只有当整套责任逻辑都错位时，才整条重写。

### 4. 商务条款

- 涉及价格、结算比例、付款周期、成果范围、排他性、独家性、赔偿上限等商务边界时，要先看是否得到明确授权。
- 无明确授权时，优先批注提示风险，并提供建议方向，而不是直接替客户拍板。

## 五、对 DOCX 修订效果的要求

- 追求“看起来像律师真的一条条改过”，而不是“机器把整段换掉了”。
- 删除类问题，应优先显示为删除原词句。
- 补位类问题，应优先显示为插入新句或新短语。
- 替换类问题，应尽量压缩替换范围，避免把整个自然段都改成修订痕迹。
- 若一条替换会在审阅视图里炸出十几个低价值碎修订，应主动并块，保留 2 到 6 组更易读的核心修改痕迹，而不是把标点和单字差异全部单独展示。
- 批注类问题，应说明风险原因和客户需要补充/确认的内容，不要只写抽象结论。
- 当前执行器已补上两层收敛：同一 run 内短语优先局部修，跨 run 优先做段落内片段修订；若复杂长条款仍拆出过多碎片，则会自动收敛为更少的可读修订块。复核时仍应重点查看极复杂长条款。

### 5.1 默认对外交付形态：修订批注一体版

- 默认只对外交付一个审阅版 DOCX，不再让用户在“纯批注 / 纯修订 / 修订批注”之间自行选择。
- 该版本应同时包含：
  - 对确定性问题的直接修订
  - 对留空项、事实待补项、授权不明商务项的批注提示
  - 对重大直接修订的必要解释性批注
- 这样既保留可直接使用的修订结果，也保留律师的风险说明和谈判提示。

### 5.2 哪些直接修订后还应保留批注

- P0 / P1 的重大风险条款，在直接修订后，原则上仍可保留简短批注，说明修改原因和风险背景。
- 纯格式、错别字、编号、标点、日期表述统一这类轻微技术性问题，通常只修订，不必再留批注。
- Alpha/智合类产物中值得借鉴的，不是“所有地方都加批注”，而是“重大问题修订后仍有解释，轻微问题直接改掉不打扰阅读”。
- 当前默认批注模板应尽量贴近律师审阅表达：`风险等级 / 风险点 / 条款位置 / 说明 / 修改建议 / 可选建议措辞`，而不是技术执行语言。

## 六、与当前技能的配合方式

1. 先读 `review-framework.md`，识别风险属于宏观 / 中观 / 微观哪一层。
2. 再读 `contract-routing.md` 与 `priority-clauses.md`，确定类型与优先审查顺序。
3. 进入对应 `contract-types/` 主文件完成条款级判断。
4. 最后用本文件决定每个问题应当：
   - 仅写入意见书
   - 直接修订
   - 局部删减
   - 局部补入
   - 整条重写
   - 仅作批注保留

---

## 七、当前默认口径

- 当前 skill 的执行取向仍然是“确定性问题优先直接修订”，但不再把“识别出的每个问题”都直接落到 Word。
- 但修订动作内部应遵循“局部删减 / 局部补入优先于整段重写”。
- 对留空项、事实待补项、授权不明的商务取舍项，默认保留批注。
- 对纯优化型、程序型、说明型问题，默认只进入审查意见书。
- 对准确的法条引用和解释性表述，默认不为精简而删减。
