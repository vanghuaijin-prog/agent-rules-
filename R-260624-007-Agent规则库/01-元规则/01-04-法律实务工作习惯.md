# 01-04 法律实务工作习惯

## 案例笔记整理规则

> 2026-05-25 建立。

**核心原则：法院精彩论述必须充分保留**
- 对"法院认为"、"裁判观点"、"本院认为"、"律师点评"等核心段落，必须**原文保留**，不可删减概括
- 法院的论证逻辑、价值判断、裁判理由、政策考量等精华表述，是判例笔记最有价值的 material
- 整理判例时，优先提取并原样保留这些段落；宁可保留冗余，不可丢失精华
- 此规则优先级高于"简洁"原则——法院原文不可为简洁而牺牲

**处理方式：**
1. 识别判例中的核心段落
2. 原文完整提取，放在笔记显著位置
3. 必要时在原文前后加注解读/批注，但原文本身不删不改
4. 精彩程度不亚于核心段落的法院附带论述（obiter dicta），同样保留

## 当事人信息 OCR 优先提取

> 2026-06-14 建立。身份证图片不靠 AI 肉眼读——精度不够。默认走百度 OCR 身份证结构化接口。

**标准流程：**
1. 调用 `~/Desktop/ocr_idcard.py auto <图片路径>`
2. 提取返回 JSON 中的 `parsed` 字段 → 直接填入文书当事人信息段落
3. 不肉眼读图猜字

**批量场景：** `ocr_idcard.py batch <文件夹> -o results.json`

## PDF/图片证据 OCR 工具链

> 2026-06-23 建立。工具：PyMuPDF (fitz) + 百度 OCR 通用文字识别。

**标准流程（三段式）：**
1. **判断 PDF 类型：** `fitz.open()` → `page.get_text()` 检测每页有无文字层
2. **文本型 PDF：** `page.get_text()` 直接提取 → 结构化整理
3. **扫描件/图片型 PDF：** `page.get_pixmap(dpi=150)` 渲染 → 百度 OCR 通用文字识别 → 输出结构化文本

**环境现状：**
- `pip3 install PyMuPDF` 已装（1.26.5）；`pdfplumber` 已装
- `requests` 未装，网络调用统一用标准库 `urllib.request`

## 行政起诉状格式模板

> 2026-05-26 建立。

**Markdown 结构**：frontmatter → 标题 → 当事人信息 → 诉讼请求 → 事实与理由 → 法律依据 → 证据清单 → 结语 → 此致法院 → 具状人 + 日期

**Word 排版**：大标题黑体 16pt 居中；章节标题黑体 14pt；子标题黑体 12pt 加粗；正文仿宋 12pt 首行缩进 2 字符；法条引用仿宋 11pt 左缩进 1cm；行距 1.25 倍；页边距上下 2.54cm 左右 3.18cm

**文件命名**：`{客户简称}诉{被告机关}-{文件类别}-v{N}.ext`

## 案件分析固定框架

所有案件分析优先采用：
1. 案件事实
2. 时间证据表
3. 争议焦点
4. 救济路径
5. 最优救济策略

必要时增加：举证责任、程序风险、实体风险、对方抗辩预测、裁判倾向

## 案件复利沉淀机制

> 2026-06-10 建立。把每次案件处理转化为可复用的 case knowledge base。

**默认动作：** 当用户提供具体案件、裁判文书、类案材料或要求办理某一案件时，AI 在完成当次任务的同时，应主动将可沉淀内容整理进一个专门的案件沉淀文档。

**沉淀字段：**
1. 案件标识：案由、案号、法院、裁判日期、案件类型、来源
2. 事实情况：核心事实、时间线、证据节点、当事人关系
3. 法律问题：争议焦点、请求权/抗辩路径、程序问题、举证责任
4. 法律依据：法规名称、条文号、条文要点、是否已元典核对
5. 类案检索：元典检索式、核心类案、反向案例、相似度、来源链接
6. 裁判规则：从元典真实案例中提炼法院裁判逻辑、裁判要旨、可迁移规则、适用边界
7. 实务启发：对诉讼策略、证据组织、文书写作、风险预判的启发
8. 可复用标签：业务领域、法律关系、关键词、相似案件链接

**整理原则：**
- 不是简单摘要，而是把案件转成可检索、可引用、可迁移的 rule + fact pattern
- 案件沉淀不能只沉淀本案事实和法条；必须用元典检索相关案例
- 对灰区业务，案件沉淀必须额外整理"合法模式 vs 无效/违法模式"的边界表
- 裁判观点必须来自真实检索案例，标明案号、法院、裁判日期、案由、裁判要旨和来源链接；不得凭经验编造"类案规则"
- 法院精彩论述仍按"案例笔记整理规则"原文保留
- 法律依据仍必须元典核对，不得把文书引用的法条默认当真

## 法律分析偏好

- 强调实务可落地
- 强调法院真实裁判逻辑，不停留在法条背诵
- 要分析法官、律师真正关注的问题
- 要分析证据链闭合
- 要分析程序推进路径
- 要分析执行可能性

## 法律文书格式规则

> 2026-05-30 建立。

### 标点符号
- 法律文书必须全中文标点：， 。 ： ； （） ""、 —— 严禁英文标点混入
- 生成后检查确认无残留英文标点

### 标题风格
- 避免 AI 程式化标题——过度对称、模板化措辞、深度嵌套编号
- 接近真实律师写作：精当有力，不修饰过度
- 写完标题自问：执业 10 年的律师会这样写吗？不会就重写

### 数字金额格式
- 法律文书数字金额不加千位分隔符（逗号），写"68000元"而非"68,800元"
- 小数点后保留两位不受影响（如 20800.00）

### 当事人信息格式
- 法律文书最上方的原告、被告、申请人、被申请人、第三人等当事人信息必须加粗
- 实例——**原告**：张三，男，1990年1月1日出生，汉族，住北京市朝阳区XX路XX号，身份证号：110101199001010000

### Word 文书样式
- 严禁使用 Word 内置蓝色"标题1/Heading 1"等一眼 AI 味的标题样式
- **强制执行规则：生成 docx 时，必须显式清空文档模板中全部 Heading 1-9 样式、Heading 1-9 Char 样式、TOC Heading 样式的 theme_color（设为 None），并将 font.color.rgb 设为 RGBColor(0,0,0)。不依赖 Word 默认样式不染色的假设。**

### 附件结构
- 统一为：附件一（证据材料）+ 附件二（法律依据）
- 法律依据不能只列法规名称和条号：短条文（小于50字）必须写完整条文；长条文（大于50字）可以不写全文，但必须写关键节选
- 附件标题必须写"附件一：证据材料""附件二：法律依据"

### 证据清单格式
- 法律文书中的证据清单默认采用法院式六列表格：编号｜证据名称｜份数｜页数｜原件或复印件｜证明对象
- 表格标题为"证据清单"，居中加粗
- "证明对象"必须写成完整事实句，格式为"证明……"
- 每项证据应标明份数、页数及原件/复印件

### 监督求助文书不得自行定性
> 2026-06-09 建立。向纪委监委、检察院等监督部门提交的求助书/举报书，本质是提供证据线索和事实线索，而非作出法律裁决。

- 所有对他人行为违法性的确定性判断，一律改为"涉嫌"
- ✗ "该行为违反…" → ✓ "该行为涉嫌违反…"
- 违禁词黑名单：保护伞、滥用职权、滥权行为、系统性违法、视而不见 → 删除或用描述性语言替代

### 法律依据和类案必须元典核对，不得默认真实
> 2026-06-09 建立法律依据核对；2026-06-18 扩展为"法律依据 + 类案裁判规则"双核对。

- 收到任何含法律依据的材料，必须用元典逐条核对
- 写任何法律文书时，除元典核对法律依据外，还必须用元典检索相关案例
- 类案检索至少覆盖：同案由/相近案由、核心争议焦点、关键事实模式、上级法院或近三年优先
- 核对四要素：法规名称是否正确 + 条文号是否存在 + 条文内容是否与原文一致 + 法规是否现行有效
- 重点怀疑对象：部门规章、规范性文件（当事人最常编造这一层），其次是司法解释

### 任何数字和事实必须可追溯，不确定信息必须留空
> 2026-06-15 建立。

**铁律四条：**
1. **全面追溯原则：** 凡是写入文书的具体信息——账号、金额、日期、身份证号、以及任何事实性陈述——写入前必须能回答"这个信息来自哪个原始材料？"。不能回答 = 不能写
2. **不确定信息留空，不补全：** "以银行查询为准"就是"以银行查询为准"，绝不能自动升级成具体数字
3. **区分原文与推理：** 文书中的陈述必须能区分（a）直接来自原始材料的事实 和（b）基于事实的法律推理/论证
4. **自检：** 每次生成文书后，逐项过一遍关键数字和事实性陈述，确认每个都有原始材料出处

### 不编造证据材料的完整定义
- 法律文书中的**所有具体信息**（数字、事实性陈述、证据名称、当事人行为描述、时间线节点等），必须有原始材料支撑
- 不得从既有文书中照抄证据清单和数字，逐一核验原始材料
- 无法确认的证据/数字/事实，标注"待补"或直接不列
- 此规则优先级高于"让文书看起来完整"——宁可留白，不可编造

## 合同审查方法

审查不仅看条款合法性，还必须分析：
1. 合同真实目的
2. 项目背景
3. 商业结构
4. 权责是否匹配
5. 风险是否可控
6. 验收是否可执行
7. 付款条件是否锁定
8. 违约责任是否对等
9. 是否存在隐性义务
10. 是否存在监管风险

输出方式：条款级修改意见 + 风险级分析 + 商业逻辑分析 + 补充协议建议

## AI 合规分析层次

**第一层：模型层**
关注：训练数据、版权、数据来源、算法安全、深度合成、模型备案、生成式 AI 监管

**第二层：应用层**
关注：用户交互、AI 陪伴、内容生成、风险提示、未成年人、虚假信息、用户依赖性

**第三层：商业层**
关注：商业模式是否合法、用户激励机制、是否涉嫌传销、是否形成非法资金池、是否形成类金融结构、平台责任边界
