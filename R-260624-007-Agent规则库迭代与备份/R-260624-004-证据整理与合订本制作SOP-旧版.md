# 证据整理与合订本制作 SOP

> **适用范围**：手上有一堆零散的诉讼证据材料（图片、PDF、聊天记录截图、Excel、音视频等），需要从"乱文件夹"状态整理到编号清晰、映射完善、按证明组合并的法院级证据合订本。
>
> **本SOP是「诉讼案件分析SOP」的Phase 2（证据整理）的详细展开版，可以独立使用，也可以嵌入全流程。**

---

## 一、在整体诉讼流程中的位置

完整的诉讼立案准备分为六个阶段：

| Phase | 名称 | 核心产出 |
|-------|------|----------|
| Phase 1 | 事实与证据 | 证据分类编号、映射表、PDF合订本 ← **本SOP覆盖这里** |
| Phase 2 | 法律研究 | 案例检索 + 法条分析备忘录（内部文件） |
| Phase 3 | 金额计算 | 诉讼请求计算明细表 |
| Phase 4 | 起诉状 | 民事起诉状.docx |
| Phase 5 | 证据清单 | 六列法院格式表格（从映射表自动生成） |
| Phase 6 | 程序性文件 | 身份证、委托书、工商登记信息等 |

本SOP只覆盖Phase 1，但它产出的映射表和合订本是后续所有阶段的基石。

---

## 二、输入与前置条件

### 2.1 输入

一个文件夹，里面装着当事人给你的各种原始材料：图片（身份证照片、银行回执截图、聊天截图）、PDF（销售数据表）、文档（客户说明）、音视频（录屏、电话录音）等。文件名通常是 hash 值或无意义的名字。

### 2.2 前置条件

- 已知道本案的基本当事人信息（原告、被告、第三人）
- 已知道大致的诉讼请求方向（要钱、要物、要行为）
- Python 环境（Pillow、pypdf、pypdfium2 库）
- Excel/LibreOffice（用于制作映射表）

## 三、四级证据编号体系

### 3.1 编号格式

```
[角色代码]-[大类编号]-[条目编号][-子项字母]
```

示例：`P1-02-03-a` = 原告1 → 财务转账凭证 → 第3份 → 子项a

### 3.2 角色代码

| 代码 | 含义 | 说明 |
|------|------|------|
| P1, P2, P3... | Plaintiff（原告） | 多个原告时递增编号 |
| D1, D2, D3... | Defendant（被告） | 多个被告时递增编号 |
| T1, T2... | Third party（第三人） | 如有第三人 |
| L | Lawyer（律师工作成果） | AI产出的分析、清单等 |
| M | Mapping（映射表） | 证据映射表本身 |
| **Q** | **Queue（待补充队列）** | 临时收纳区，见§3.5 |

### 3.3 大类编号（以原告方为例）

| 编号 | 类别名称 | 典型内容 |
|------|----------|----------|
| 01 | 当事人身份材料 | 身份证、委托书、营业执照 |
| 02 | 财务转账凭证 | 银行回执、转账记录、流水 |
| 03 | 收据押金凭证 | 收据、押金条 |
| 04 | 发票 | 增值税发票、普通发票 |
| 05 | 沟通记录 | 微信聊天截图、语音、视频录屏 |
| 06 | 销售数据 | 销售报表、提成计算表 |
| 07 | 客户说明 | 当事人出具的情况说明 |
| 08 | 被告书面确认 | 被告方出具的欠条、说明、确认函 |

> **⚠️ 易错点：类别划分要按证据性质，不是按"看起来是什么"**。比如当事人给你一张"说明"的截图，你可能会习惯性放进05-沟通记录。但如果这份"说明"是被告正式出具的书面文件（有签名、有日期、有金额确认），它的性质是**被告书面确认**，应该单独建类（08），因为它在证明力上远强于聊天记录。
>
> **判断标准**：这份材料是"对话过程中的截取"还是"一方正式出具的书面文件"？前者归沟通记录，后者归书面确认。

### 3.4 文件命名规范

```
[证据ID]-[可读描述].[扩展名]
```

示例：
- `P1-01-01-辛丽伟身份证.jpg`
- `P1-02-01-建设银行回执-仁康→正和-21185.69.jpg`
- `P1-08-01-说明-欠款28338.66.png`

> **⚠️ 易错点：文件名必须可读，带金额、主体、用途。** 原始 hash 文件名（如 `09c739367fde9fa54d824853daf13312.jpg`）在整理阶段必须改掉。以后查找时全靠文件名，hash名 = 没名字。
>
> **命名原则**：看到文件名就知道这是什么、涉及多少钱、谁给谁的。金额精确到分。箭头 `→` 表示资金流向。

### 3.5 Q-Queue（待补充队列）

在根目录建一个 `Q-Queue/` 文件夹。当事人说"我还有个东西要补"或者你发现缺证据时，新材料先丢进 Q-Queue，**不急着编号**。

等收集齐了、确认要用了，再按正常流程：分类 → 编号 → 命名 → 移入正式目录 → 更新映射表。

> **为什么不在收到时立刻编号？** 因为很多补充材料最后可能用不上（重复的、模糊的、和案件无关的）。提前编号会导致编号跳号或者映射表反复修改。Q-Queue 是缓冲区，进了正式序列才算数。

## 四、文件夹结构搭建

### 4.1 根目录结构

```
{案件名}/
├── P1-原告({姓名})/
│   ├── 01-当事人身份材料/
│   ├── 02-财务转账凭证/
│   ├── 03-收据押金凭证/
│   ├── 04-发票/
│   ├── 05-沟通记录/
│   ├── 06-销售数据/
│   ├── 07-客户说明/
│   ├── 08-被告书面确认/
│   └── ...（按需增加）
├── D1-被告({公司名})/
├── T1-第三人/（如有）
├── L-律师工作成果/
│   └── 02-证据合并输出/（合订本PDF放这里）
├── M-证据映射表/
│   └── M-00-证据映射表.xlsx
└── Q-Queue/（待补充材料缓冲区）
```

### 4.2 操作步骤

1. 在根目录下创建上述文件夹
2. 遍历原始文件夹中的所有文件
3. 逐份阅读内容，判断类别
4. 重命名为可读文件名（带证据ID前缀）
5. 移入对应子文件夹
6. 检查根目录已清空（除文件夹外无散落文件）

> **⚠️ 易错点：先读再分，不要先建太多空文件夹。** 大类编号应该根据实际材料来定——如果当事人没给你发票，就不需要建04-发票。空文件夹没用，反而暗示"这里应该有东西"导致混淆。
>
> **例外**：01-当事人身份材料几乎一定会有，可以先建。其他的等材料决定了再建。

## 五、证据映射表制作

### 5.1 映射表是什么

映射表是整个体系的"中枢神经"。它用一张 Excel 表格把"按文书类型分类的物理文件"和"按证明目的分类的逻辑分组"连接起来。同一份证据可以同时出现在多个证明组中。

### 5.2 七列结构

| 列名 | 说明 | 示例 |
|------|------|------|
| 证据ID | 四级编号 | P1-02-01 |
| 文件名 | 可读文件名（不含ID前缀，因为ID列已有） | 建设银行回执-仁康→正和-21185.69 |
| 来源 | 谁提供的 | 原告 |
| 待证事实（简） | 一句话简写 | 证明货款流向 |
| 证明对象（完整版） | 完整的法院级事实句 | 证明2023年X月X日，仁康医药通过建设银行向通化正和转账21185.69元，系销售回款 |
| 证明组 | 属于哪些证明组 | A, C |
| 备注 | 简练的补充说明 | 仁康医药→通化正和 |

### 5.3 证明对象（完整版）写法

**硬规则**：完整事实句，包含时间 + 主体 + 金额/行为 + 法律意义。

好的写法：
> 证明2023年4月至2024年1月间，原告通过仁康医药向被告通化正和共计转账13503.54元，系内蒙九州通药品销售回款。

不好的写法：
> 转账记录 ❌（太简）
> 银行回执 ❌（没说证明什么）

### 5.4 证明组设计

每个证明组对应一个你需要在法庭上证明的核心事实。用字母代号，映射表数据区下方附索引：

| 证明组 | 待证事实 |
|--------|----------|
| **A** | 劳动/代理关系存在 |
| **B** | 押金已交付10000元 |
| **C** | 提成款金额为17338.66元 |
| **D** | 被告未返还款项 |
| **E** | 原告积极主张权利 |

> **⚠️ 易错点：证明组字母的含义必须在映射表里直接可见。** 不要只放在第二个 sheet 里——人翻Excel时容易忘，拉到表底下一眼就看到最方便。建议数据区下方空一行，直接贴索引表。

### 5.5 跨组引用的主次原则

同一份证据可能同时证明多个事实。比如"说明"文件（P1-08-01）同时证明押金交付（B）、提成金额（C）、被告未返还（D）。

**主次原则**：
- **主要证明目的**：放在最核心的证明组 PDF 中（实物页面）
- **次要证明目的**：在证据清单中写"详见组X第X号证据"，不重复合并 PDF

> **⚠️ 易错点：不要在多个组的PDF里重复放同一份证据的页面。** 法官翻合订本时会困惑"这不是刚才看过的吗"。一次出现、多处引用。

### 5.6 关键证据标记

对案件有决定性作用的证据，在映射表中用**红色加粗 + ★** 标记。

典型的关键证据：被告方出具的书面确认文件（欠条、说明、对账单），因为它同时证明了多个待证事实且证明力极强。

### 5.7 备注风格

备注要**简练精当**，写人话，不要写成判决书。好的备注：

| 场景 | 好的备注 | 不好的备注 |
|------|----------|------------|
| 两个主体之间的转账 | 仁康医药→通化正和 | 该证据显示仁康医药有限责任公司通过建设银行向通化正和医药有限公司进行了转账 |
| 内容重复 | 与01-03重复 | 该证据的内容与本文件夹中编号为P1-01-03的证据存在内容上的重叠 |
| 需要后续处理 | 需查看内容后归类 | 待进一步审查该文件之具体内容以确定其在本证据体系中之分类归属 |
| 被其他证据替代 | 已由08-01替代 | （同上啰嗦版） |

## 六、文件格式处理规则

不是所有文件都能直接合并进 PDF。以下是处理规则：

| 文件类型 | 处理方式 | 谁来做 |
|----------|----------|--------|
| JPG / PNG 图片 | 直接合并 | AI |
| PDF（A4尺寸） | 直接合并 | AI |
| PDF（非A4尺寸） | 渲染为图片→放A4页→合并 | AI |
| Excel (.xlsx) | **截图保存为图片** → 合并 | **当事人/律师手动** |
| Word (.docx) | **导出为 PDF** → 合并 | **当事人/律师手动** |
| 视频 (.mp4) | **无法直接合并。** 截屏关键画面为图片→合并。或刻盘/U盘单独提交 | **当事人/律师手动** |
| 音频 (.mp3等) | **转文字稿→Word→PDF→合并** | **当事人/律师手动** |
| 长图拼接截图 | 已拼接好的长图PDF，直接合并 | AI |

> **⚠️ 易错点：遇到非 PDF 友好格式时，提示当事人手动处理，不要硬转。** 比如 AI 把 Excel 直接转 PDF，排版会乱；AI 把 Word 硬转 PDF，字体可能不对。让当事人自己截图/导出更靠谱。
>
> **音频/视频处理原则**：以后尽量让当事人提供截图而非视频。视频的信息密度远低于截图 + 文字说明。如果已经有视频了，提取关键帧截图作为替代。

## 七、PDF合并与页码标注

### 7.1 合并策略

按证明组分别合并，最后拼合订本：

```
证据合并输出/
├── 证据组A-{待证事实名}.pdf    （组A的证据合并）
├── 证据组B-{待证事实名}.pdf    （组B的证据合并）
├── ...
└── 证据合订本-{案件名}.pdf     （全部拼起来）
```

### 7.2 每个组内的证据排序

按叙事逻辑排序，不是按编号排序。法官读证据是顺着故事线读的。

典型叙事顺序：
1. 建立关系（委托书、合同）
2. 履行行为（转账、交货、提供服务）
3. 违约事实（未付款、未返还）
4. 金额确认（对账单、说明）
5. 权利主张（催款记录、律师函）

> **⚠️ 易错点：P1-01-01 不一定是组内第一页。** 编号是分类编号，不是叙事顺序。组内排序应根据证明逻辑决定。

### 7.3 双层页码标注

每一页叠加两个信息：

| 位置 | 内容 | 作用 |
|------|------|------|
| **底部居中** | 第X页（连续页码） | 全局定位，法官说"翻到第15页"大家都能找到 |
| **右上角** | 证据ID（如 P1-02-01） | 身份标识，对应映射表 |

> **⚠️ 易错点：页码是图层叠加，不是修改原文件。** 不管底下是截图、扫描件、长图拼接还是文字PDF，页码都能加。原理是用 Pillow 或 pypdf 在已有页面上叠加一层文字。

### 7.4 技术实现（Python）

**依赖库**：Pillow（图片→A4页）、pypdfium2（非A4 PDF→图片渲染）、pypdf（PDF合并）

**关键参数**：
- DPI = 150（标准清晰度，可提到200-300）
- A4尺寸 = 1240 × 1754 像素（150DPI下）
- 中文字体：`/System/Library/Fonts/STHeiti Medium.ttc`（macOS）

**处理逻辑**：
1. 图片文件 → Pillow打开 → 缩放适配A4（留边距） → 白色背景上贴图 → 叠加页码 → 存为PDF页
2. A4尺寸的PDF → pypdf直接读取 → 叠加页码层
3. 非A4尺寸的PDF → pypdfium2渲染为图片（150DPI） → 按图片流程处理
4. 跳过极小页面（<200px），通常是空白页脚碎片

> **⚠️ 易错点1：非A4 PDF直接合并会出空白页。** 用 `pypdf.merge_transformed_page()` 对非标准尺寸的PDF页（比如从Excel导出的699×768像素页）会丢失内容。必须先用 pypdfium2 渲染为图片，再按图片处理。
>
> **⚠️ 易错点2：极小页面要跳过。** 有些PDF的第2页只有122×74像素（基本是空白的页脚碎片），不跳过会产生一个空白页。设置 `min_pixels = 200` 阈值过滤。
>
> **⚠️ 易错点3：中文引号在 Python heredoc 中会冲突。** 如果用 bash heredoc 写 Python 脚本，中文引号 `""` 可能和 Python 字符串分隔符冲突。用 unicode 转义 `\u201c\u201d` 或者改用单引号。

### 7.5 DPI与清晰度

默认150 DPI足够法庭使用（相当于标准打印质量的75%）。如果证据包含小字或细节需要放大查看，可以提到200甚至300 DPI，只需改脚本里的 DPI 常量，重新跑一遍即可。

> 150 DPI → 文件大小适中，21页合订本约3-5MB
> 200 DPI → 更清晰，文件约5-8MB
> 300 DPI → 印刷级，文件约10-15MB

### 7.6 输出存放位置

证据合并输出文件夹放在 `L-律师工作成果/02-证据合并输出/` 里，因为合订本本身是律师工作产物，不应散放在根目录。

## 八、证据清单生成

### 8.1 生成时机

**PDF合订本定稿后再生成，不做草稿再回填。**

理由：页码取决于最终PDF排版，先出清单留空白页码 = 后面逐项回填 = 工作量翻倍。映射表里已有所有数据字段，从映射表自动生成六列表格是一次到位的。

唯一例外：如果团队需要先审阅证据清单的文字表述（名称、证明对象写得对不对），可以先出一版页码栏写"见合订本第__页"的草稿。但正常情况下，后做更省事。

### 8.2 法院六列格式

| 编号 | 证据名称 | 份数 | 页数 | 原件或复印件 | 证明对象 |
|------|----------|------|------|------------|----------|
| 1 | 辛丽伟身份证 | 1 | 1 | 复印件 | 证明原告身份... |

### 8.3 跨组证据的处理

在次要证明组的证据清单行中，"页数"列写"详见组X第X号证据"，不需要重复计算页数。

## 九、常见陷阱与应对

### 9.1 证据金额与诉求金额不一致

**典型场景**：整理证据过程中发现一份"说明"文件，确认的总金额比起诉状写的多（比如多了退货保证金1000元）。

**应对**：立即标记差异，提醒律师核实是否需要修改诉讼请求金额。不要默默改掉——起诉状金额变更可能影响管辖、诉讼费计算、保全金额。

### 9.2 分类错误

**典型场景**：把被告正式出具的书面文件放进了"沟通记录"。

**应对**：见§3.3 的判断标准——"对话过程的截取" vs "正式出具的书面文件"。发现分错后立即建新类、移文件、更新映射表。

### 9.3 编号跳号

**典型场景**：删除了一份证据后，编号不连续。

**应对**：允许跳号，不改号。编号是身份标识，不是序号。改一个号需要更新映射表、证据清单、合订本页码，牵一发动全身。删除的证据在映射表中标记"已删除"即可，编号保留空缺。

### 9.4 新证据插入

**典型场景**：当事人补充了新证据。

**应对**：
1. 新材料先进 Q-Queue
2. 确认要用后，分配新编号（在对应大类下递增）
3. 命名、移入正式目录
4. 更新映射表（加行、更新证明组包含列表）
5. 重新生成合订本PDF（页码会自动重算）

### 9.5 销售数据PDF页面异常

**典型场景**：从Excel导出的销售数据PDF，页面尺寸不是A4（可能是699×768、631×294等奇怪尺寸）。

**应对**：必须用 pypdfium2 渲染为图片后按图片流程处理。直接 pypdf 合并会出空白页。详见§7.4。
